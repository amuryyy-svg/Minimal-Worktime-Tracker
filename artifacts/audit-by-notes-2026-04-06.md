## Детальный разбор по пунктам

### 1. Сброс данных в приложении выдает ошибку

Статус: не подтвердил статически, но реализация хрупкая и требует runtime-проверки.

Что есть в коде:
- `src/renderer/app.js:1050` содержит `clearStoredData()`
- `src/main.js:358` содержит IPC `data:clear`
- в main-процессе очищается только папка автобэкапов
- сами записи приложения чистятся в renderer через `localStorage.removeItem(...)` в `src/renderer/app.js:441`

Вывод:
- явной синтаксической причины для падения не видно
- но очистка размазана между renderer и main, нет одного атомарного сценария
- ошибки удаления в main глушатся через `console.warn`, а IPC всё равно возвращает `{ cleared: true }`
- из-за этого легко получить ситуацию "очистка завершена частично, UI считает, что всё хорошо"

Приоритет: высокий, потому что пользователь уже видел сбой.

### 19. Почему приложение ест ~50 МБ RAM и весит ~700 МБ, не переписать ли движок?

Статус: сам вопрос понятен, но причина в основном не в конкретном баге.

Что я проверил локально:
- `release/Minimal Worktime Tracker 1.1.2.exe` примерно `74.5 MB`
- `release/Minimal Worktime Tracker Setup 1.1.2.exe` примерно `74.7 MB`
- вся папка `release/` сейчас занимает около `149.3 MB`
- весь репозиторий занимает около `698.5 MB`
- из них `node_modules` занимают около `548.9 MB`

Вывод:
- цифра около `700 MB` у меня подтверждается для всей рабочей папки проекта, а не для shipping binary
- baseline RAM в районе `50 MB` для маленького Electron-приложения звучит реалистично, а не аномально
- статический аудит не даёт оснований считать, что приложение уже нужно переписывать с Electron на другой стек
- обсуждать переписывание имеет смысл только если будут подтверждены реальные product-проблемы: медленный cold start, высокий idle CPU, утечки памяти, плохой tray / autostart UX

Приоритет: низкий.

## Дополнительные находки вне самих заметок

### 1. Основной renderer-файл слишком перегружен для следующей фазы фич

`src/renderer/app.js` сейчас занимает `1393` строки и смешивает:

- локальное хранилище
- таймер
- расчёты календаря и streak
- UI-рендер
- settings UI
- backup/import/export flows
- интеграцию с Electron bridge

Для текущего объёма это ещё терпимо, но пункты `11-14` в этот файл добавлять опасно. Перед аналитикой и интервалами лучше отделить:

- state/persistence layer
- timer/session domain layer
- UI-панели
- desktop bridge handlers

### 2. Runtime icon и build icon разведены по разным источникам

Это важно, потому что иначе очень легко "чинить не ту иконку":

- tray и runtime окно используют `Logo.png`
- сборка использует `build/icon.ico`

Без унификации пайплайна проблема из заметок `2-3` будет возвращаться.

### 3. Покрытие тестами не закрывает реальные desktop-регрессии

Сейчас тестируется только `tracker-core`, а реальные жалобы пользователя относятся к:

- tray
- autostart
- clear data
- восстановлению состояния таймера после перезапуска

Именно поэтому пункты `1`, `6`, `15` могли дойти до заметок.

## Рекомендуемый порядок работ

1. Сначала закрыть реальные баги поведения: `6`, `15`, затем воспроизвести и добить `1`.
2. Затем добрать correctness по времени и дням: `7`, `16`, `17`.
3. Отдельным этапом решить, нужен ли большой redesign модели данных под `11-13`.
4. Только после этого браться за аналитику / отдельное окно / помодоро-уведомления: `12`, `14`.
5. UI-полировку `2`, `3`, `8`, `9`, `10` делать после фикса поведенческих багов.

## Итог

По коду видно, что проект уже покрывает базовый сценарий таймера, календаря, streak, tray и backup, но из заметок подтверждаются два настоящих баг-кластера:

- рассинхрон автозагрузки
- автозапуск таймера после старта приложения

Самый большой продуктовый долг не в UI, а в модели данных: пока приложение хранит только дневные суммы, пункты про интервалы, аналитику и ручную правку времени останутся дорогими и хрупкими.

## Название функции

Feature 1. Артефакты

## Суть функции

Функция "Артефакты" добавляет в оффлайн-трекер второй слой прогресса поверх учёта времени:

- таймер и календарь продолжают отвечать на вопрос "сколько работал"
- артефакты отвечают на вопрос "что реально появилось в результате работы"

Артефакт в рамках MVP этой функции:
- это один локально сохранённый скриншот
- привязанный к конкретному дню
- с необязательными полями `название` и `описание`
- отображаемый как визуальное доказательство прогресса на главном экране и на отдельной странице дня

## Почему функция вводится

Исходная продуктовая задача:
- усилить мотивацию не только фиксацией времени, но и фиксацией реального результата
- разбить длинную работу на ощутимые промежуточные этапы
- дать пользователю локальную, приватную, не-социальную среду фиксации прогресса

Функция не является:
- социальной сетью
- публичной лентой
- системой аккаунтов или облачной синхронизации

Функция является:
- локальным журналом промежуточных результатов работы
- визуальным таймлайном прогресса по дням
- оффлайн-слоем мотивации внутри уже существующего трекера

## Предусловия функции

Функция должна реализовываться поверх уже существующей базы приложения после Фаз `1-4` и `6`.

Это означает:
- day/time model уже работает на persisted формате с версиями и миграцией
- ручная правка времени, day details и текущий desktop shell уже существуют
- import/export уже есть как пользовательские сценарии и должны быть расширены, а не сломаны
- приложение по-прежнему остаётся single-window desktop app без сетевой зависимости

## Цель функции

После реализации функции пользователь должен уметь:

- быстро добавить скриншот текущего результата через `drag & drop` или `Ctrl+V`
- сохранить его даже если не захотел сразу писать название и описание
- видеть под таймером до трёх последних артефактов выбранного дня
- открыть отдельную страницу дня с "полкой" артефактов
- открыть конкретный артефакт как карточку/фотографию с метаданными
- редактировать название и описание уже сохранённого артефакта
- удалять артефакт
- хранить всё локально и включать артефакты в backup/export/import

## Что входит в функцию

- локальное хранение артефактов без сети и аккаунтов
- добавление артефакта через `drag & drop`
- добавление артефакта через вставку из буфера `Ctrl+V`
- форма ввода `названия` и `описания` после добавления
- сохранение артефакта даже при закрытии формы без заполнения полей
- домашний блок артефактов под таймером
- отдельный экран/страница выбранного дня с полкой артефактов
- просмотр метаданных артефакта
- редактирование метаданных артефакта
- удаление артефакта
- локальное фоновое сжатие изображений
- отдельная стратегия более агрессивного сжатия старых артефактов
- включение артефактов в clear data, backup, export и import
- исторический режим просмотра дня, в котором главный таймер показывает выбранный день, но не позволяет запуск

## Что не входит в функцию

- любая социальная механика
- публикация артефактов наружу
- облачная синхронизация
- мультипользовательский режим
- комментарии, лайки, реакции, подписки
- массовая загрузка пачки изображений
- поддержка видео, PDF, Figma-ссылок и других типов файлов в MVP
- отдельные маркеры артефактов в календарной сетке в MVP
- OCR, AI-описания и автоматическая классификация артефактов

## Ограничения функции

- приложение остаётся полностью оффлайн
- нельзя добавлять новые npm-зависимости без отдельного согласования
- нельзя блокировать UI во время декодирования, ресайза, сжатия и записи изображения
- нельзя хранить исходные полноразмерные скриншоты "на всякий случай", если цель MVP именно экономия места
- нельзя ломать существующие сценарии `clear day`, `clear data`, `import/export`, `autobackup`, `streak`, `manual-adjustment`
- нельзя допускать, чтобы артефакты смешивались с time entries как единый источник истины по рабочему времени

## Выбранные трактовки спорных решений

### 1. Функция остаётся полностью локальной

Принятое правило:
- артефакты живут только на устройстве пользователя
- никакого сервера, логина, обмена между пользователями и облачной синхронизации в MVP нет

Следствие:
- хранение изображений и метаданных проектируется как локальное persisted state + локальные файлы

### 2. Артефакт привязывается к выбранному дню

Принятое правило:
- по умолчанию выбранным днём является `сегодня`
- если пользователь выбрал другой день, новый артефакт сохраняется в этот выбранный день

Следствие:
- отдельного шага "к какому дню привязать" в форме создания не нужно
- день назначения определяется UI-контекстом

### 3. Выбор не-сегодняшнего дня переводит главный таймер в исторический режим

Принятое правило:
- при выборе дня, отличного от сегодняшнего, центральный таймер показывает итог выбранного дня
- таймер визуально становится слегка приглушённым/серым
- старт/пауза/запуск для такого дня недоступны

Дополнительная трактовка:
- если реальный таймер сегодня уже идёт, он продолжает идти для `today`
- выбранный исторический день не перехватывает runtime timer
- пользователь смотрит историю, а не перепривязывает живой таймер на прошлую дату

### 4. В MVP за одно действие добавляется только один скриншот

Принятое правило:
- `Ctrl+V` добавляет один артефакт
- `drag & drop` в MVP принимает только первое валидное изображение из дропа

Следствие:
- если пользователь дропнул несколько файлов, система берёт первый поддерживаемый image file и явно игнорирует остальные с кратким сообщением

### 5. Закрытие формы метаданных не отменяет сохранение артефакта

Принятое правило:
- если пользователь уже вставил или дропнул скриншот, артефакт всё равно сохраняется
- при закрытии формы без подтверждения поля `title` и `description` остаются пустыми строками

Следствие:
- форма после добавления не является "транзакцией create/cancel"
- она является только удобной возможностью сразу заполнить метаданные

### 6. Артефакты можно редактировать и удалять позже

Принятое правило:
- у уже сохранённого артефакта можно изменить `title` и `description`
- артефакт можно удалить с удалением его файла с диска

### 7. Отдельных маркеров артефактов в календаре в MVP нет

Принятое правило:
- наличие артефактов в MVP показывается через зону под таймером и через экран-полку дня
- в календарной сетке не добавляются специальные бейджи, точки или значки

Пояснение:
- это удерживает scope в разумных границах
- визуальный сигнал и так появляется при выборе дня

### 8. На главном экране показываются максимум три карточки

Принятое правило:
- на стартовом экране в зоне артефактов отображаются не более трёх последних артефактов выбранного дня
- полный список отображается только на отдельной странице-полке

### 9. Изображения хранятся не в оригинале, а в оптимизированной локальной форме

Принятое правило:
- исходный скриншот после нормализации не хранится как отдельный full-size original
- в persisted storage входит только оптимизированная версия для просмотра и компактный thumbnail

Выбранный ориентир для MVP:
- display/full версия: JPEG, максимум `500px` по большей стороне
- thumbnail версия: отдельная уменьшенная карточечная копия

Пояснение:
- пользовательский сценарий не про пиксель-perfect архив исходников
- он про узнаваемый след работы при разумном размере на диске

### 10. Старые артефакты можно дожимать сильнее

Принятое правило:
- артефакты старше `90` дней (`3+ месяца`) могут быть фоново пережаты более агрессивно
- это делается только в фоне и не должно блокировать приложение

Пояснение:
- для старых артефактов главная ценность в узнаваемости и факте наличия
- сверхточное качество для них менее критично

### 11. `clear day` и `clear data` ведут себя по-разному

Принятое правило:
- `clear selected day time` очищает только рабочее время дня и не удаляет артефакты
- `clear data` удаляет и time data, и артефакты, и файлы артефактов

Пояснение:
- артефакты не являются time entries
- нельзя неожиданно стирать визуальный журнал только из-за очистки времени дня

### 12. Просмотр артефакта оформляется как фотокарточка

Принятое правило:
- детальный просмотр артефакта оформляется как карточка/фотография в рамке
- при открытии используется переворот "лицевая сторона -> обратная сторона" как основной UI-приём

Fallback правило:
- при `prefers-reduced-motion: reduce` или при нестабильности анимации интерфейс должен показывать тот же контент без принудительного 3D-переворота

### 13. Import остаётся replace-сценарием, а не merge-сценарием

Принятое правило:
- import нового backup bundle заменяет текущую локальную базу приложения целиком
- merge существующих локальных артефактов с импортируемыми в MVP не поддерживается

Пояснение:
- это сохраняет текущую модель import как восстановление состояния, а не сложную синхронизацию двух библиотек
- это упрощает работу с файловым хранилищем и не создаёт конфликтов ID и dangling files

### 14. Сбой image pipeline не должен оставлять битый черновик

Принятое правило:
- если decode, resize, encode или запись файла завершились ошибкой, готовый артефакт не создаётся
- пользователь получает понятное сообщение об ошибке
- metadata entry и файлы-огрызки не должны оставаться в state или на диске

Пояснение:
- функция должна либо завершить сохранение целиком, либо аккуратно откатиться
- пустая карточка без изображения в MVP не считается допустимым результатом ошибки

## UX-модель функции

### Главный экран

Под центральным таймером появляется отдельная зона артефактов выбранного дня.

Состояния зоны:

1. Empty state
- пунктирная рамка
- понятная подпись в духе "Перетащи скриншот сюда или вставь через Ctrl+V"
- визуальный фокус на том, что это место для фиксации результата

2. Drag-over state
- усиливается свечение
- появляется анимация готовности принять файл
- пунктирная рамка и фон явно показывают, что дроп валиден

3. Filled state
- до трёх карточек/превью
- композиция не сеткой, а лёгким веером/стопкой, ближе к референсу из пользовательского описания
- основной акцент остаётся у таймера, артефакты не должны визуально его съедать

### Экран дня "Полка артефактов"

При клике на блок артефактов выбранного дня или на одну из карточек открывается отдельный экран дня.

Требования:
- это экран внутри текущего окна приложения, а не новый `BrowserWindow`
- таймер остаётся видимым в нижней части интерфейса
- если открыт не сегодняшний день, таймер в историческом режиме серый и read-only
- на экране показываются все артефакты выбранного дня
- порядок отображения: newest first

### Детальный просмотр артефакта

При клике на конкретный артефакт открывается детальный просмотр.

Содержимое:
- изображение в рамке
- название
- описание
- дата дня
- точное время добавления
- действия `Редактировать` и `Удалить`

UI-приём:
- основное визуальное решение это фотокарточка, которая может "перевернуться" и показать обратную сторону с подписью

## Поддерживаемые пользовательские сценарии

### Сценарий 1. Добавление скриншота через paste

1. Пользователь находится в приложении.
2. Выбран какой-то день.
3. Пользователь нажимает `Ctrl+V`, если в буфере есть изображение.
4. Приложение создаёт новый артефакт для выбранного дня.
5. Открывается форма с `названием` и `описанием`.
6. Если пользователь сохраняет поля, они записываются.
7. Если пользователь закрывает форму, артефакт всё равно остаётся, но поля пустые.

### Сценарий 2. Добавление скриншота через drag & drop

1. Пользователь перетаскивает изображение на зону артефактов.
2. Зона переходит в drag-over state.
3. После дропа создаётся новый артефакт для выбранного дня.
4. Открывается форма метаданных.
5. Дальнейшее поведение совпадает со сценарием paste.

### Сценарий 3. Добавление артефакта в прошлый день

1. Пользователь выбирает исторический день в календаре.
2. Таймер переключается на этот день и становится read-only.
3. Пользователь вставляет или дропает скриншот.
4. Артефакт сохраняется в выбранный исторический день.

### Сценарий 4. Просмотр всех артефактов дня

1. Пользователь открывает полку выбранного дня.
2. Видит полный список артефактов за этот день.
3. Может открыть конкретный артефакт и посмотреть метаданные.

### Сценарий 5. Редактирование

1. Пользователь открывает детальный просмотр артефакта.
2. Меняет `название` и/или `описание`.
3. Сохраняет.
4. Обновление сразу отражается и на главной, и на полке дня.

### Сценарий 6. Удаление

1. Пользователь открывает детальный просмотр.
2. Нажимает `Удалить`.
3. Подтверждает действие.
4. Метаданные и соответствующие image files удаляются.

## Целевая persisted-модель данных

Функция меняет persisted format, потому что появляются новые пользовательские сущности и связь с локальным файловым хранилищем.

Принятое решение:
- persisted state повышается до `v9`
- time entries продолжают жить отдельно
- артефакты не смешиваются с `days[dateKey].entries`

Предлагаемая структура:

```json
{
  "version": 9,
  "days": {
    "2026-04-12": {
      "entries": []
    }
  },
  "artifactsByDay": {
    "2026-04-12": ["artifact_01"]
  },
  "artifacts": {
    "artifact_01": {
      "id": "artifact_01",
      "dayKey": "2026-04-12",
      "createdAtMs": 1775940600000,
      "updatedAtMs": 1775940600000,
      "title": "Первый список карточек",
      "description": "Собрал каталог и фильтры на главной.",
      "source": "paste",
      "storage": {
        "fullRelativePath": "artifacts/2026/04/artifact_01.jpg",
        "thumbRelativePath": "artifacts/2026/04/artifact_01.thumb.jpg",
        "mime": "image/jpeg",
        "width": 500,
        "height": 281,
        "sizeBytes": 18432,
        "compressionTier": "fresh"
      }
    }
  },
  "settings": {},
  "timerState": {}
}
```

## Инварианты модели

- `artifactsByDay[dayKey]` содержит только ID реально существующих артефактов
- каждый артефакт принадлежит ровно одному `dayKey`
- `title` и `description` могут быть пустыми строками
- `createdAtMs` фиксирует момент добавления артефакта
- `updatedAtMs` обновляется при редактировании метаданных
- persisted image format в MVP это только `image/jpeg`
- для готового артефакта должны существовать `fullRelativePath` и `thumbRelativePath`
- `compressionTier` в MVP принимает `fresh | archived`
- артефакты не влияют на расчёт рабочего времени, streak и daily totals

## Локальное файловое хранение

Файлы артефактов должны храниться не в localStorage, а в отдельной директории внутри app data.

Предлагаемая структура:

- `<userData>/artifacts/YYYY/MM/<artifactId>.jpg`
- `<userData>/artifacts/YYYY/MM/<artifactId>.thumb.jpg`

Требования:
- renderer не работает с произвольными путями напрямую
- запись/удаление файлов идёт через тонкий IPC boundary
- persisted state хранит только относительные пути, а не произвольные абсолютные пользовательские пути

## Image pipeline

### Вход

Поддерживаемые источники в MVP:
- `paste` из clipboard image
- `drop` image file

Минимально поддерживаемые форматы входа:
- PNG
- JPEG/JPG
- WebP

Невалидные входы:
- текст без изображения
- неподдерживаемые бинарные файлы
- пустой clipboard

### Обработка

Image pipeline должен быть асинхронным и неблокирующим.

Минимальный pipeline:
1. принять изображение
2. декодировать его
3. создать thumbnail
4. создать display/full версию
5. перекодировать в JPEG
6. сохранить обе версии на диск
7. записать метаданные в persisted state

### Параметры хранения

`full`:
- максимум `500px` по большей стороне
- JPEG
- качество подбирается эмпирически, но так, чтобы скриншот оставался легко узнаваемым и не выглядел испорченным

`thumbnail`:
- отдельная уменьшенная версия для карточек и полки
- JPEG
- размер достаточно малый для быстрого отображения без заметного смазывания карточек

### Фоновое архивное сжатие

Для артефактов старше `90` дней:
- разрешается фоновая перекодировка `full` в более агрессивный JPEG
- качество можно снижать сильнее, если артефакт остаётся узнаваемым
- job должен идти постепенно, без массовой синхронной обработки всех файлов сразу

### Цель по занимаемому месту

Это не жёсткая гарантия, а инженерный ориентир:
- свежие артефакты должны быть разумно компактными
- старые артефакты должны уметь ужиматься ещё сильнее
- порядок объёмов должен оставаться бытовым и не пугать пользователя ростом на диске

## Логика таймера и выбора дня

### Сегодня

Когда выбран `today`:
- центральный таймер работает как обычно
- можно запускать и останавливать таймер
- артефакты добавляются в сегодняшний день

### Не сегодня

Когда выбран день, отличный от `today`:
- центральный таймер показывает итог выбранного дня
- визуально таймер приглушён
- start/pause недоступны
- блок артефактов и полка относятся к выбранному дню

### Если таймер сегодня реально идёт

Когда runtime timer идёт по сегодняшнему дню, а пользователь смотрит прошлый день:
- live timer не останавливается
- выбранный исторический день не получает running-state
- UI явно остаётся в режиме просмотра истории

## Резервные копии, import и export

Так как теперь есть бинарные файлы, plain JSON backup уже недостаточен для полного экспорта.

Принятое решение:
- новый полный backup/export формат должен стать архивным bundle-форматом
- внутри bundle лежат:
  - `manifest.json` с persisted state
  - директория `artifacts/` с image files

Требования:
- import нового bundle должен полностью восстанавливать и время, и артефакты
- import старого JSON backup без артефактов должен оставаться рабочим
- export без артефактов не считается полным экспортом приложения после внедрения функции
- auto backup должен включать артефакты, а не только JSON state
- import bundle работает как полная замена текущего локального состояния, а не как merge

## Clear data и migration

### Migration

Переход на функцию требует:
- миграции persisted state с `v8 -> v9`
- старая `v8` структура без артефактов должна корректно подниматься как пустая по блоку артефактов

### Clear data

`clear data` обязан:
- удалить persisted state
- удалить локальное хранилище изображений артефактов
- не оставлять orphaned files

### Clear selected day time

`clear selected day time`:
- очищает только time data
- не удаляет артефакты дня

## Функциональные требования

## FR-1. Empty state зоны артефактов

Когда у выбранного дня нет артефактов, под таймером должна отображаться зона с:
- пунктирной рамкой
- понятным CTA на `drop` и `Ctrl+V`
- визуально заметным, но не агрессивным свечением

## FR-2. Drag & drop

При перетаскивании валидного изображения:
- зона артефактов должна анимированно переходить в активное состояние
- drop должен создавать новый артефакт выбранного дня

Если файлов несколько:
- берётся первый валидный image input
- остальные игнорируются

## FR-3. Paste

Когда фокус приложения позволяет глобальную вставку:
- `Ctrl+V` с изображением должен создавать новый артефакт выбранного дня

Ограничение:
- если пользователь уже печатает внутри поля `title`/`description`, paste работает как обычная вставка в текстовое поле и не должен создавать новый артефакт

## FR-4. Форма метаданных после добавления

После успешного приёма изображения:
- открывается форма с полями `название` и `описание`
- поля необязательные
- закрытие формы не откатывает сохранение изображения

## FR-5. Карточки на главном экране

На главном экране:
- отображаются максимум `3` артефакта выбранного дня
- карточки визуально организованы как лёгкая стопка/веер, а не как тяжёлая сетка
- клик по блоку открывает полный экран дня

## FR-6. Экран-полка выбранного дня

Отдельный экран дня должен:
- показывать все артефакты выбранного дня
- не открываться в новом окне
- сохранять нижний таймерный блок в интерфейсе

## FR-7. Детальный просмотр

При открытии артефакта пользователь видит:
- изображение
- название
- описание
- дату дня
- время добавления
- кнопки `Редактировать` и `Удалить`

## FR-8. Переворот фотокарточки

Детальный просмотр должен поддерживать визуальный паттерн "фото в рамке -> переворот -> подпись на обороте".

Допустимое упрощение:
- при reduced motion или техническом fallback контент остаётся тем же, но без обязательной 3D-анимации

## FR-9. Редактирование

Пользователь может:
- изменить `title`
- изменить `description`

Редактирование не должно:
- создавать новый артефакт
- перезаписывать изображение заново

## FR-10. Удаление

Удаление артефакта должно:
- требовать подтверждения
- удалять metadata entry
- удалять image files с диска
- сразу обновлять все связанные UI-поверхности

## FR-11. Исторический режим таймера

Выбор не-сегодняшнего дня должен:
- переводить таймер в read-only historical mode
- визуально блокировать запуск
- при этом не ломать текущий running timer сегодняшнего дня, если он есть

## FR-12. Backup/import/export

После внедрения функции:
- полный export обязан включать артефакты
- import полного bundle обязан восстанавливать артефакты
- старые backup-файлы без артефактов обязаны продолжать импортироваться

## FR-13. Background compression

Сжатие и архивная пережатка:
- не должны блокировать renderer UI
- должны выполняться в фоне
- должны быть устойчивы к перезапуску приложения

## Ожидаемое техническое решение

## 1. Доменный слой

Нужен отдельный доменный слой для артефактов, а не логика россыпью по `app.js`.

Минимально ожидается выделение отдельного модуля, например:
- `src/renderer/artifacts-core.js`

Ответственность:
- normalize/sanitize artifact metadata
- selectors по `artifactsByDay`
- создание/редактирование/удаление metadata records
- валидация dayKey, title, description, compression tier

## 2. Persistence boundary

Нужно расширить storage boundary, например через:
- обновление `tracker-storage.js`
- или отдельный вспомогательный слой под artifact manifest

Ответственность:
- миграция `v8 -> v9`
- чтение/запись persisted metadata
- подготовка manifest для backup/export

## 3. Файловый слой

Файловые операции не должны расползтись по `main.js`.

Минимально ожидается отдельный main-side helper, например:
- `src/artifact-file-store.js`

Ответственность:
- запись JPEG файлов
- удаление файлов
- очистка artifact directory
- подготовка bundle export/import

## 4. Image processing

Нужен отдельный async pipeline для:
- decode
- resize
- encode
- thumbnail generation

Если обработка делается в renderer:
- она должна быть вынесена из UI wiring
- результат файловой записи всё равно проходит через main/preload boundary

Если используется worker:
- у worker должен быть явный протокол сообщений
- отмена и cleanup должны быть определены явно

## Ожидаемые файлы изменений

Минимально ожидаемый набор:
- `src/renderer/app.js`
- `src/renderer/index.html`
- `src/renderer/styles.css`
- `src/renderer/tracker-storage.js`
- `src/renderer/tracker-core.js` или отдельный `src/renderer/artifacts-core.js`
- `src/preload.js`
- `src/main.js`
- отдельный helper для artifact-file IO на стороне Electron main
- тесты под storage/domain/import/export
- обновление `ARCHITECTURE.md`

Минимально ожидаемое покрытие:

2. Metadata normalization

5. Import/export

## Ручные сценарии

Обязательный smoke перед принятием:

1. Вставить скриншот через `Ctrl+V`, закрыть форму без текста, убедиться что артефакт сохранился.
2. Перетащить файл мышью, увидеть drag animation и успешное сохранение.
3. Выбрать прошлый день, добавить туда артефакт, убедиться что таймер серый и не запускается.
4. Открыть экран дня, увидеть полный список.
5. Открыть детальную карточку, проверить переворот или reduced-motion fallback.
6. Отредактировать название и описание.
7. Удалить артефакт.
8. Сделать export, затем import, убедиться что картинки и metadata вернулись.
9. Выполнить `clear selected day time`, убедиться что артефакт не исчез.
10. Выполнить `clear data`, убедиться что исчезли и metadata, и файлы.

## Критерии приёмки

Функция считается принятой, если:

- артефакты действительно работают оффлайн без сети
- paste и drag & drop поддерживаются
- закрытие формы не теряет уже добавленный скриншот
- главный экран показывает до трёх артефактов выбранного дня
- прошлые дни поддерживаются через historical timer mode
- экран-полка выбранного дня работает в том же окне
- детальный просмотр даёт доступ к метаданным
- edit/delete работают
- backup/export/import включают артефакты
- clear data очищает и state, и файлы
- сжатие идёт в фоне и UI не подвисает

## Риски и меры

### Риск 1. UI начинает тормозить на больших скриншотах

Мера:
- decode/resize/encode делать асинхронно
- не держать тяжёлую обработку внутри UI event handler без отложенного pipeline

### Риск 2. Orphaned files на диске

Мера:
- file deletion и clear data делать через централизованный file-store helper
- после delete/import/clear прогонять reconciliation metadata vs files

### Риск 3. Backup становится слишком сложным

Мера:
- сразу принять bundle-формат как основной экспорт после внедрения артефактов
- не пытаться запихнуть binary images в старый JSON-only контракт

### Риск 4. Исторический режим таймера запутает пользователя

Мера:
- визуально явно отличать interactive today timer от read-only historical timer
- не прятать признак живого running-state, если сегодня таймер реально идёт

### Риск 5. Scope расползётся в мини-DAM систему

Мера:
- в MVP поддерживать только изображения
- только один screenshot per add
- без публичности, без тегов, без поиска, без соц-механик

## Итоговая трактовка функции

Feature 1 "Артефакты" в рамках этого ТЗ трактуется не как медиа-галерея и не как соцфункция.

Это:
- локальный визуальный журнал прогресса
- привязанный к дням работы
- встроенный в текущий таймер и календарный workflow
- сохраняющий оффлайн-след результатов работы даже тогда, когда проект ещё не закончен

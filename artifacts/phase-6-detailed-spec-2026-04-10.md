## Название фазы

Фаза 6. Desktop UX и визуальная полировка

## Предусловие фазы

Фаза 6 должна опираться как минимум на принятую и влитую Фазу 3.

Это означает:
- новая архитектурная база renderer уже существует
- tray/autostart/cold start и data reset уже стабилизированы
- persisted format уже версионирован и не должен меняться хаотично

Фаза 6 может идти параллельно с Фазой 4, но только при жёстком соблюдении границ:
- Фаза 6 не должна менять interval model, timer domain и backup semantics
- допустимые конфликты с Фазой 4 ожидаемы в `app.js`, `index.html`, `styles.css`, но не должны расползаться на `tracker-storage` и `tracker-timer`
- если для dark theme потребуется новый persisted setting, это должно быть минимальное добавление в `settings`, а не новая ветка доменной логики

## Цель фазы

Довести до ума desktop-оболочку приложения после того, как базовая логика уже стала надёжной:

- сделать tray/icon pipeline технически правильным и визуально чистым
- привести сворачивание и возврат окна к одному предсказуемому правилу "живёт в tray, пока пользователь явно не вышел"
- добавить тёмную тему без побочных эффектов для календаря, overlays и custom titlebar
- добавить лёгкую анимацию активного таймера без нагрузки и без вмешательства в business logic
- привести tray menu к состоянию, где из него можно нормально управлять приложением и таймером

## Что входит в фазу

- отдельный runtime pipeline для tray assets
- state-aware tray icon для состояний `idle` и `running`
- корректное поведение hide-to-tray не только через кастомную кнопку, но и через window events
- пересборка tray menu под более понятный desktop flow
- тёмная тема приложения как пользовательская настройка
- лёгкая motion-полировка активного таймера
- минимальная визуальная полировка настроек только в объёме, который нужен для theme toggle и согласованного вида

## Что не входит в фазу

- отдельное окно настроек
- отдельное окно аналитики
- изменение interval model, manual-adjustment model или timer persistence
- новые уведомления, помодоро-звуки и action buttons из Windows notifications
- новая always-visible индикация из свежих заметок
- расследование пункта про `50mb RAM / 700mb` как отдельная performance phase
- переписывание приложения с Electron на другой стек
- большой редизайн основного окна поверх уже выбранного визуального языка

## Выбранные трактовки спорных решений

### 1. Packaging icon и runtime tray icon — это разные задачи

Принятое правило:
- `build/icon.ico` отвечает за exe/installer иконку
- tray icon не строится из него и не ресайзится на лету из большого логотипа

Выбранный подход:
- для tray используются отдельные small-size assets, подготовленные специально под Windows tray sizes
- assets для tray живут в runtime-доступном пути внутри пакуемых файлов приложения

Пояснение:
- задача из заметок про "зазубрины" не решается простым `resize()` большого PNG
- чёткая мелкая иконка требует отдельного pixel-fitted asset pipeline

### 2. Пункт про "сворачивание в tray" трактуется как единое hide-to-tray поведение

Принятое правило:
- пользовательский выход из окна не должен завершать приложение, пока пользователь явно не выбрал `Exit`

Следствие:
- custom minimize
- window minimize event
- close action окна

должны приводить к согласованному hide-to-tray сценарию, а не к трем разным формам поведения.

### 3. Пункт про "пауза/старт" в tray не вводит новый persisted paused state

Принятое правило:
- Фаза 6 не создаёт отдельное третье состояние таймера поверх существующих `running/stopped`

Выбранная трактовка:
- в tray menu пользователь видит desktop control таймера
- когда таймер идёт, действие формулируется как `Pause`
- когда таймер остановлен, действие формулируется как `Start`

Технический смысл:
- `Pause` в рамках этой фазы маппится на уже существующую безопасную stop/start semantics
- то есть таймер завершает текущий интервал и потом может быть запущен снова с нового момента

Пояснение:
- это закрывает пользовательскую потребность "управлять таймером из tray"
- и не превращает Phase 6 в отдельную переработку timer domain

### 4. Тёмная тема в Фазе 6 делается как явная настройка приложения, а не как большой appearance framework

Принятое правило:
- Фаза 6 добавляет только минимально достаточную пользовательскую тему

Выбранный подход:
- в persisted settings добавляется новое поле `theme`
- допустимые значения: `light`, `dark`
- default для существующих пользователей: `light`

Пояснение:
- это минимальный безопасный вариант без расширения области фазы до `system/light/dark` orchestration
- отдельная поддержка `system` может быть добавлена позже, если реально понадобится

### 5. Анимация активного таймера должна быть CSS-first и low-cost

Принятое правило:
- animation polish не должен зависеть от новых JS intervals, requestAnimationFrame loops или постоянных DOM recalculation

Выбранный подход:
- анимация строится на CSS-классах активного состояния
- она работает только в состоянии `timer running`
- при `prefers-reduced-motion: reduce` анимация отключается или упрощается до статического выделения

### 6. Визуальная полировка настроек ограничивается интеграцией dark theme

Принятое правило:
- Фаза 6 не является редизайном settings overlay и не переносит настройки в отдельное окно

Следствие:
- разрешены только те UI-правки, которые нужны, чтобы настройка темы выглядела нативно и не ломала текущую структуру
- пункт про отдельное окно настроек остаётся в орбите Фазы 5 и отдельно принимаемого продуктового решения

## Целевой пользовательский результат

После Фазы 6 пользователь должен уметь:

- свернуть приложение и быть уверенным, что оно осталось жить в tray
- открыть или скрыть окно из tray predictably
- запускать и "ставить на паузу" учёт времени из tray без возврата в основное окно
- переключить приложение в тёмную тему
- визуально различать idle и active состояние таймера без навязчивой анимации
- видеть аккуратную tray icon без грязных краёв и мыла на типовых Windows scale

## Функциональные требования

## FR-1. Dedicated runtime tray icon pipeline

Приложение должно перестать использовать один большой `Logo.png`, который просто ресайзится в runtime.

Требование:
- для tray должны быть подготовлены отдельные small-size assets
- минимально поддерживаемые размеры: `16`, `20`, `24`, `32` px
- должны существовать отдельные варианты как минимум для состояний `idle` и `running`

Допускается:
- использовать один визуальный мотив логотипа

Не допускается:
- рассчитывать на runtime resize большого исходника как на финальное решение качества

## FR-2. Tray icon должен корректно обновляться по состоянию таймера

Когда таймер:
- остановлен, tray icon показывает idle state
- запущен, tray icon показывает running state

Обновление иконки должно происходить:
- при старте таймера
- при остановке таймера
- при восстановлении состояния UI после hide/show

Иконка не должна зависеть от light/dark темы самого окна приложения.

## FR-3. Hide-to-tray должен быть единым правилом поведения окна

Пользовательские действия:
- нажатие на кастомную кнопку сворачивания
- системное `minimize`
- нажатие на кнопку закрытия окна

должны вести к согласованному результату:
- окно скрывается
- приложение остаётся живым
- tray продолжает работать
- taskbar presence обновляется предсказуемо

Явный выход из приложения должен оставаться отдельным действием через tray menu `Exit`.

## FR-4. Tray menu должен стать полноценной desktop-control поверх текущего timer flow

Минимальный состав tray menu после Фазы 6:
- `Show` или `Hide` в зависимости от текущей видимости окна
- `Start` или `Pause` в зависимости от текущего состояния таймера
- `Exit`

Допустимо добавить разделитель и переставить порядок пунктов, если это повышает ясность.

Не требуется:
- отдельная команда `Resume`
- отдельная persisted pause-state модель

Поведение:
- `Start` запускает текущий timer flow
- `Pause` завершает текущий running segment тем же безопасным способом, которым сейчас работает остановка таймера

## FR-5. Тёмная тема должна быть пользовательской настройкой и переживать перезапуск

В настройках приложения должен появиться явный control выбора темы:
- `Light`
- `Dark`

Требование:
- выбранная тема сохраняется в пользовательских settings
- после перезапуска приложения тема восстанавливается
- смена темы применяется без перезапуска

## FR-6. Тёмная тема должна покрывать все основные поверхности приложения

После переключения в dark theme корректно работают и визуально читаются:
- основная оболочка окна
- titlebar и кнопки окна
- status/streak/target strip
- таймерная панель
- календарь
- выбранный день
- settings overlay
- confirm overlay
- toast
- hover/focus/active состояния controls

Не допускается:
- текст с недостаточным контрастом
- исчезающие границы
- невидимые hover/focus состояния
- белые "дыры" в overlay/панелях при переключении темы

## FR-7. Активный таймер получает лёгкую анимацию

Когда таймер запущен:
- у timer panel появляется лёгкая пульсация или аналогичный аккуратный motion cue
- анимация не должна быть агрессивной, большой по амплитуде или отвлекающей
- animation не должна влиять на layout и размеры соседних элементов

Когда таймер остановлен:
- анимация полностью пропадает

При reduced motion:
- анимация либо отключается, либо заменяется статическим highlight

## FR-8. Build/runtime contract по иконкам должен быть явным

Должно быть явно разведено:
- что используется для packaging icon
- что используется для runtime tray icon

Требование:
- packaged build содержит runtime tray assets
- dev run и packaged run используют одну и ту же логику поиска runtime tray assets
- изменение packaging icon не должно неявно ломать tray icon и наоборот

## FR-9. Изменения Фазы 6 не должны менять доменную семантику трекера

После внедрения Фазы 6 не должны измениться:
- способ расчёта времени дня
- import/export semantics
- interval model
- streak rules
- правила clear data / clear day / autostart / cold start

Если для темы добавляется новый persisted setting, он не должен менять логику старых полей settings.

## Техническое решение

### 1. Tray assets

Рекомендуемая структура:

```text
src/
  assets/
    tray/
      idle-16.png
      idle-20.png
      idle-24.png
      idle-32.png
      running-16.png
      running-20.png
      running-24.png
      running-32.png
```

Принцип:
- `build/icon.ico` остаётся packaging asset
- runtime tray assets кладутся в `src/**`, чтобы гарантированно попасть в packaged files по текущему `package.json`

В `main` должен появиться небольшой слой выбора tray asset по:
- app state (`idle/running`)
- доступному размеру / representation

### 2. Main process tray behavior

В `main` должна быть централизована логика:
- загрузки runtime tray icon
- переключения иконки при обновлении timer state
- нормализации hide-to-tray поведения по `close` и `minimize`
- построения tray menu через отдельный helper, а не через размазанный inline template

Если `src/main.js` начинает разрастаться, допускается вынести чистые helpers:
- icon path resolution
- tray menu label resolution

Но не нужно выносить половину main-процесса в большой универсальный service.

### 3. Theme setting

В renderer/domain слое должен появиться новый setting:

```json
{
  "theme": "light"
}
```

Минимально требуемая поддержка:
- default value
- sanitize invalid values
- persistence
- UI wiring в settings overlay

Рекомендуемое место изменения:
- `tracker-core` для default/sanitize/settings normalization
- `app.js` для UI wiring и применения атрибута темы
- `styles.css` для токенов и theme selectors

### 4. Theme application strategy

Рекомендуемый подход:
- на корневой оболочке приложения ставится `data-theme="light"` или `data-theme="dark"`
- CSS variables переопределяются через `[data-theme="dark"]`

Преимущество:
- это сохраняет текущий styling approach на CSS custom properties
- и не требует переписывать компоненты на другой styling model

### 5. Timer motion

Рекомендуемый подход:
- анимация должна сидеть в CSS
- активное состояние задаётся существующим JS-state/class wiring
- motion должен быть ограничен timer panel, status dot или близким акцентным элементом

Не рекомендуется:
- запускать отдельный JS-таймер для визуального pulse
- делать непрерывную перерисовку через DOM style mutation

### 6. Settings UI scope

Разрешённые изменения:
- добавить theme selector
- уточнить spacing, grouping и visual emphasis внутри overlay, если это нужно для согласованности

Неразрешённые изменения:
- превращать Фазу 6 в редизайн settings IA
- выносить настройки в отдельное окно

## Обязательные ручные сценарии

### Сценарий 1. Tray icon в dev run

- запустить приложение через `npm start`
- убедиться, что idle tray icon чёткая на типовом масштабе Windows
- запустить таймер
- убедиться, что icon меняется на running-state

### Сценарий 2. Hide-to-tray

- открыть окно
- нажать кастомную кнопку сворачивания
- убедиться, что окно исчезло, приложение осталось жить в tray
- вернуть окно кликом по tray
- повторить через window minimize event
- повторить через close button и убедиться, что это не `Exit`

### Сценарий 3. Tray menu timer control

- открыть tray menu в idle состоянии
- увидеть `Start`
- запустить таймер из tray
- снова открыть tray menu и увидеть `Pause`
- остановить таймер из tray
- убедиться, что время учлось так же, как при обычной кнопке в окне

### Сценарий 4. Dark theme

- переключить тему на `dark`
- пройти основные поверхности приложения
- перезапустить приложение
- убедиться, что тема восстановилась

### Сценарий 5. Packaged build

- собрать packaged build
- проверить tray icon, hide-to-tray и theme persistence не только в dev, но и в packaged приложении

## Критерии приёмки

- tray icon визуально чистая на стандартных Windows scale как минимум `100%`, `125%`, `150%`
- runtime tray icon и packaging icon разведены технически и не зависят от ресайза `Logo.png`
- сворачивание и закрытие окна работают предсказуемо как hide-to-tray, пока пользователь явно не выбрал `Exit`
- tray menu позволяет управлять таймером без открытия окна
- тёмная тема переживает перезапуск и не ломает читаемость интерфейса
- motion active timer существует, но не отвлекает и не создаёт дополнительной логической сложности
- основные сценарии трекера не изменили свою доменную семантику

## Риски и меры

Риск:
- Фаза 6 пойдёт слишком глубоко в timer semantics из-за формулировки "Pause"

Мера:
- заранее зафиксировать, что в рамках этой фазы `Pause` — это desktop wording поверх существующего stop/start flow, без нового persisted state

Риск:
- tray assets окажутся чёткими только в dev, но разъедутся в packaged build

Мера:
- проверять packaged build как обязательный сценарий приёмки

Риск:
- dark theme вскроет десятки low-contrast состояний в overlay и calendar UI

Мера:
- строить тему не ad hoc, а через системные CSS variables и пройти все основные surface states вручную

Риск:
- параллельная разработка с Фазой 4 создаст конфликт в `app.js` и `styles.css`

Мера:
- держать ownership Фазы 6 на tray/main/theme/styling, не лезть в domain changes Фазы 4 без необходимости

Риск:
- попытка "чуть-чуть улучшить настройки" раздует scope до нового окна и отдельной IA

Мера:
- явно оставить отдельное окно настроек вне scope Фазы 6

## Итоговое определение готовности

Фаза 6 считается завершённой, если:
- приложение стало выглядеть и вести себя как аккуратное desktop tray-приложение
- у пользователя появились тёмная тема и ясный tray control таймера
- при этом код не уехал в новую доменную сложность и не сломал groundwork Фаз 1-4

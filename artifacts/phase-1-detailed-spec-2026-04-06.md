## Название фазы

Фаза 1. Стабилизация критичных багов поведения

## Цель фазы

Устранить три пользовательски значимых дефекта, которые уже подтверждены аудитом и подрывают доверие к приложению:

- рассинхрон автозагрузки между интерфейсом и фактическим состоянием Windows
- автозапуск таймера после холодного запуска приложения
- нестабильный и неатомарный сценарий полного сброса данных

## Пункты заметок, закрываемые фазой

- `1. Сброс данных в приложении выдает ошибку`
- `6. Автозагрузка включена, но после перезапуска приложения и в настройках ее нет, а в диспетчере есть`
- `15. Сейчас запустил приложение и таймер автоматически запустился, что неверно`

## Что входит в фазу

- исправление стартовой инициализации состояния приложения
- исправление взаимодействия renderer ↔ main для настройки автозагрузки
- переработка полного сброса данных в подтверждаемый сценарий "всё очищено / очистка не выполнена"
- добавление минимального тестового покрытия на новые правила и ручных сценариев проверки

## Что не входит в фазу

- очистка отдельного дня
- переработка streak и логики выходных
- добавление интервалов работы
- редактирование рабочего времени
- аналитика
- темная тема, анимации, иконки, tray polishing
- звуки и уведомления

## Ограничения фазы

- нельзя ломать текущий формат сохранённых данных
- нельзя вводить breaking change для импорта существующих бэкапов
- нельзя требовать новых npm-пакетов
- нельзя оставлять автозагрузку в split-brain-состоянии между renderer storage и Windows

## Выбранные трактовки спорных требований

### 1. Как трактовать "таймер не должен стартовать автоматически"

Принятое правило:
- любой холодный запуск приложения должен поднимать UI с остановленным таймером

Следствия:
- состояние `timerState.isRunning`, сохранённое в предыдущей сессии, не должно автоматически восстанавливаться как активный таймер
- suspend / resume внутри одного живого процесса продолжает работать как сейчас
- импорт бэкапа не должен запускать таймер автоматически, даже если в бэкапе `timerState.isRunning === true`

### 2. Что считать источником истины для автозагрузки

Принятое правило:
- фактическое состояние Windows является источником истины

Следствия:
- локально сохранённое `settings.autostart` является лишь кэшом UI-состояния
- при старте приложения UI обязан синхронизироваться с `app.getLoginItemSettings()`
- изменение toggle в интерфейсе должно выполняться через отдельный подтверждаемый IPC round-trip, а не через fire-and-forget событие

### 3. Что считать успешным полным сбросом данных

Принятое правило:
- успешный сброс означает, что очищены runtime app data, локальное сохранение renderer и автобэкапы приложения, а интерфейс переведён в дефолтное состояние

Следствия:
- ручные экспортированные бэкапы вне app data не удаляются
- если очистка storage не завершилась, UI не должен показывать успех
- после успешной очистки приложение не должно тут же восстановить старое состояние отложенным `scheduleSave()`

## Текущее проблемное поведение в коде

### Автозагрузка

Сейчас:
- `src/renderer/app.js` меняет `settings.autostart` локально через `setAutostart()`
- затем renderer отдельно шлёт `sendSettingsState(...)`
- `src/main.js` немедленно применяет `app.setLoginItemSettings(...)`
- при следующем старте renderer читает автозагрузку только из `localStorage`
- main при старте использует `wasOpenedAtLogin` только для скрытого старта окна, но не отправляет renderer фактическое состояние `openAtLogin`

Проблема:
- UI и система могут разъехаться
- типичный случай: ОС уже получила новое значение, а `localStorage` не успел сохраниться

### Автостарт таймера после cold start

Сейчас:
- `src/renderer/app.js` берёт `loadedState.timerState.isRunning`
- если значение `true`, то приложение считает, что таймер активен, и начинает тикать с текущего момента

Проблема:
- закрытие приложения во время активного таймера переносит "запущенное" состояние в следующий полноценный запуск
- это не соответствует ожидаемому поведению пользователя

### Полный сброс данных

Сейчас:
- main очищает только папку автобэкапов
- renderer отдельно удаляет ключи `localStorage`
- ошибки удаления в main логируются как warning, но IPC всё равно возвращает `{ cleared: true }`
- результат операции не является атомарным

Проблема:
- UI может показать успешную очистку даже при частичном сбое
- из-за разнесённости ответственности сложнее воспроизвести и стабильно починить ошибку из заметки

## Требования к реализации

## FR-1. Bootstrapping стартового состояния

При старте приложения renderer должен получать и использовать bootstrap-данные от main до финального `renderAll()`.

Минимально bootstrap-данные должны содержать:
- фактическое состояние автозагрузки в Windows
- признак запуска приложения через login item, если он нужен для скрытого старта

Ожидаемое поведение:
- UI автозагрузки показывает фактическое системное состояние
- состояние таймера на холодном старте принудительно нормализуется к `stopped`

## FR-2. Автозагрузка изменяется через подтверждаемый IPC

Переключение автозагрузки не должно больше зависеть от того, успел ли renderer сохранить `localStorage`.

Требование:
- для автозагрузки вводится отдельный IPC request/response
- main применяет новое значение через `app.setLoginItemSettings(...)`
- затем main повторно читает `app.getLoginItemSettings()` и возвращает подтверждённое значение
- renderer обновляет `settings.autostart` только по подтверждённому результату

Допустимое минимальное решение:
- generic `settings-state` остаётся для языка и прочих не-системных параметров
- автозагрузка выносится из него в отдельный канал

## FR-3. Холодный запуск всегда поднимается с остановленным таймером

Требование:
- после загрузки persisted state приложение не должно продолжать старую running-сессию

Ожидаемое правило:
- при bootstrap `timerState.isRunning` принудительно сбрасывается в `false`
- `timerState.lastTickMs` на cold start всегда `null`
- UI и tray отражают остановленное состояние

Дополнительное требование:
- после импорта бэкапа таймер также остаётся остановленным

## FR-4. Полный сброс данных выполняется как единая операция

Требование:
- общий сброс должен быть реализован как одна подтверждаемая операция с ясным итогом

В рамках фазы успешный сброс обязан:
- удалить автоматические бэкапы приложения
- очистить renderer local storage приложения
- остановить таймер в памяти
- сбросить settings и logs к дефолту
- очистить armed state кнопки сброса и связанные временные таймеры UI

В случае ошибки:
- пользователь получает ошибку, а не ложный success
- UI не должен переходить в состояние "очищено", если операция не завершилась

## FR-5. Отложенные операции не должны возрождать старое состояние после сброса

Требование:
- перед clear data нужно отменять отложенное сохранение
- нужно дождаться pending auto backup
- после успешного clear нельзя повторно записать старые данные из зависших in-memory ссылок

## FR-6. Фаза не меняет persisted format version

Требование:
- `PERSISTED_STATE_VERSION` в этой фазе не поднимать
- миграции не добавлять

Пояснение:
- фаза устраняет runtime-баги, а не меняет структуру данных

## Предлагаемое техническое решение

### Блок A. Новый bootstrap IPC из main в renderer

Добавить новый подтверждаемый bridge-метод для начальной загрузки состояния.

Минимальный контракт:

```js
{
  autostart: boolean,
  launchedAtLogin: boolean
}
```

Источник данных:
- `app.getLoginItemSettings().openAtLogin`
- `app.getLoginItemSettings().wasOpenedAtLogin`

Назначение:
- убрать инициализацию автозагрузки исключительно из `localStorage`
- дать renderer корректную стартовую картину до первого рендера

### Блок B. Выделенный IPC для изменения автозагрузки

Добавить новый bridge-метод, например:
- `desktopAPI.setAutostart(enabled)`

Ожидаемая семантика:
- renderer вызывает `invoke`
- main применяет `app.setLoginItemSettings({ openAtLogin, openAsHidden })`
- main повторно читает `app.getLoginItemSettings()`
- renderer получает подтверждённое состояние и обновляет локальный settings-state

Почему это нужно:
- текущая схема `scheduleSave() + sendSettingsState()` создаёт гонку между локальным сохранением и системным состоянием

### Блок C. Нормализация таймера на cold start

Добавить явный шаг bootstrap-нормализации:

- загруженные `logs` и `settings` сохраняются
- загруженное `timerState.isRunning` для cold start игнорируется
- runtime timer инициализируется как остановленный

Рекомендуемое место:
- рядом с `loadState()` и первичной инициализацией `timerState`

Важно:
- не путать cold start с `powerMonitor suspend/resume`, там текущая логика остаётся

### Блок D. Нормализация таймера при import backup

После импорта:
- состояние данных импортируется
- таймер всё равно устанавливается в `stopped`

Причина:
- импорт бэкапа является заменой данных, а не восстановлением живой исполняемой сессии

### Блок E. Атомарный clear data

Рекомендуемая схема:

1. renderer переводит UI кнопки в disabled
2. renderer отменяет `scheduleSave()` и дожидается `pendingAutoBackupPromise`
3. renderer вызывает подтверждаемый clear IPC
4. main:
   - удаляет автобэкапы
   - очищает storage сессии приложения через session API
   - при необходимости пересоздаёт служебные директории
   - возвращает явный успех или ошибку
5. только после подтверждённого успеха renderer:
   - сбрасывает in-memory state к default
   - рендерит чистый UI
   - показывает success toast

Для очистки storage допускается использовать `Session.clearStorageData(...)`, так как API существует в используемой версии Electron.

### Блок F. Явные ошибки вместо "успеха по умолчанию"

Требование к main IPC:
- если какой-либо критичный шаг clear data завершился ошибкой, IPC должен вернуть ошибку, а не `{ cleared: true }`

Допустимый формат:
- `throw new Error(...)`
- либо структурированный результат `{ cleared: false, reason: ... }`

Но результат должен быть однозначным для renderer.

## Ручные сценарии проверки

### Сценарий 1. Автозагрузка

1. Запустить приложение.
2. Включить автозагрузку.
3. Закрыть приложение.
4. Запустить приложение заново вручную.
5. Проверить, что toggle автозагрузки остаётся включённым.
6. Выключить автозагрузку.
7. Повторить перезапуск.
8. Проверить, что toggle остаётся выключённым.

Ожидаемый результат:
- UI соответствует `app.getLoginItemSettings().openAtLogin`

### Сценарий 2. Таймер после перезапуска

1. Запустить таймер.
2. Подождать несколько секунд.
3. Закрыть приложение.
4. Запустить приложение заново вручную.
5. Проверить, что таймер остановлен.
6. Проверить, что уже накопленное время за день не потеряно.

Ожидаемый результат:
- время дня сохранено
- активная running-сессия не восстановлена

### Сценарий 3. Полный сброс данных

1. Создать состояние с временем, настройками и автобэкапом.
2. Выполнить clear data из UI.
3. Проверить, что:
   - календарь очищен
   - настройки сброшены к дефолту
   - таймер остановлен
   - success toast показывается только при фактическом успехе

Ожидаемый результат:
- приложение приходит в чистое дефолтное состояние без ошибки

### Сценарий 4. Импорт бэкапа

1. Импортировать бэкап, где `timerState.isRunning === true`.
2. Проверить стартовое состояние после импорта.

Ожидаемый результат:
- данные импортированы
- таймер не начинает идти сам

## Критерии приёмки

Фаза считается завершённой, если одновременно выполнены все условия:

- автозагрузка в настройках совпадает с фактическим состоянием Windows после повторного запуска приложения
- ручной cold start приложения всегда показывает остановленный таймер
- импорт бэкапа не запускает таймер автоматически
- общий сброс данных отрабатывает без ложного success
- после clear data приложение находится в дефолтном состоянии
- `npm run lint` проходит
- `npm test` проходит
- если TypeScript type-check отсутствует, это явно зафиксировано как отсутствие конфигурации, а не пропущенная проверка

## Риски и меры

### Риск 1. Очистка session storage окажется шире, чем нужно

Мера:
- очищать только storage приложения
- если потребуется, ограничить очистку по session/origin, а не вычищать лишнее

### Риск 2. Смена автозагрузки создаст новый race между UI и IPC ack

Мера:
- renderer должен обновлять toggle только по подтверждённому ответу main
- на время операции контрол должен быть disabled

### Риск 3. Исправление cold start сломает существующие ожидания по восстановлению состояния

Мера:
- явно зафиксировать новое правило в коде и, при необходимости, в README
- не смешивать cold start и suspend/resume

### Риск 4. Clear data снова восстановит старое состояние отложенным сохранением

Мера:
- обязательно гасить `scheduleSave()`
- дожидаться `pendingAutoBackupPromise`
- не писать новый persist до завершения успешного reset flow

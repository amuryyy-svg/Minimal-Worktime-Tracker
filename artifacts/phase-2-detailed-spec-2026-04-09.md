## Выбранные трактовки спорных сценариев

### 1. Что означает "очистить день"

Принятое правило:
- очищается только учтённое рабочее время выбранного дня

Следствия:
- очищение дня не меняет `weekendDays`
- очищение дня не меняет `dayOverrides` автоматически
- статус дня и время остаются двумя разными пользовательскими действиями

Пояснение:
- это минимально безопасное поведение в текущей модели данных
- если пользователь хочет вернуть день в выходной, он делает это отдельным действием

### 2. Что делать, если пользователь стартует таймер в выходной

Принятое правило:
- старт таймера для текущего дня автоматически переводит этот день в рабочий через явный override `work`

Следствия:
- правило применяется только к сегодняшнему дню, а не к произвольно выбранной дате в календаре
- правило действует и для регулярного выходного, и для дня с explicit `off`
- перевод выполняется до начала накопления новой running-сессии

Пояснение:
- это соответствует тексту заметки `#16`
- это также устраняет состояние "день с рабочим временем всё ещё считается выходным"

### 3. Как streak должен относиться к историческим правкам

Принятое правило:
- `off` исключает день из streak-логики независимо от наличия `workMs`
- `work` принудительно включает день в рабочую логику даже если по календарю это выходной
- историческая правка дня должна пересчитывать streak немедленно

Следствия:
- если пользователь задним числом отмечает день как `off`, цепочка рабочих дней не должна ломаться на этом дне
- если пользователь задним числом снимает `off`, streak должен считаться заново по обычным правилам

### 4. Можно ли очищать сегодняшний день во время активного таймера

Принятое правило:
- очищение текущего дня при активном таймере не выполняется

Минимально допустимое поведение:
- UI блокирует действие
- либо показывает сообщение "остановите таймер перед очисткой дня"

Пояснение:
- это убирает двусмысленность между persisted временем и живой running-сессией
- это безопаснее, чем пытаться одновременно чистить `workMs` и держать таймер активным

## Функциональные требования

## FR-1. Очистка выбранного дня

В интерфейсе выбранного дня должно появиться отдельное действие очистки времени дня.

Требование:
- действие доступно только когда у выбранного дня есть что очищать
- перед выполнением требуется явное подтверждение
- после успешной очистки календарь, панель выбранного дня и streak пересчитываются

Успешная очистка должна:
- удалить или обнулить `logs[dateKey].workMs` для выбранной даты
- сохранить остальные дни без изменений
- не запускать полный reset приложения

## FR-2. Подтверждение очистки дня

Для очистки дня нужен отдельный защитный UX-механизм.

Допустимые реализации:
- двухшаговая кнопка по аналогии с clear data
- отдельный confirm dialog
- отдельный warning toast с коротким окном подтверждения

Обязательное свойство:
- пользователь не должен очистить день одним случайным кликом

## FR-3. Блокировка очистки активного текущего дня

Если выбран сегодняшний день и таймер сейчас запущен:
- очистка дня не выполняется
- пользователь получает явное сообщение, почему действие не выполнено

## FR-4. Старт таймера на выходном дне

Если пользователь запускает таймер сегодня, а текущий день считается выходным:
- приложение обязано поставить для сегодняшней даты `dayOverrides[today] = "work"`
- затем начать таймер
- затем пересчитать визуальное состояние календаря и streak

## FR-5. Исторические правки выходных сразу влияют на streak

Изменение статуса дня через `toggle day off` должно немедленно отражаться на:
- панели выбранного дня
- календаре
- streak

Требование:
- не должно быть сценария, когда день уже visually помечен как `off`, а streak ещё считает его рабочим барьером

## FR-6. Сочетание `workMs > 0` и `off`

На уровне текущей модели данных допустимо состояние:
- у дня есть время
- но день помечен как `off`

В этой фазе правило должно быть единым:
- для streak такой день считается выходным и пропускается
- для UI день остаётся видимым как день с временем, но со статусом выходного

Пояснение:
- это нужно для исторической коррекции, когда пользователь задним числом решает, что день не должен ломать цепочку

## FR-7. Фаза не меняет формат хранения данных

В рамках фазы:
- не добавляются интервалы
- не меняется структура `logs`
- не поднимается версия persisted state

## Предлагаемое техническое решение

### Блок A. Точечные helper-функции для работы с днём

Рекомендуется выделить небольшие pure helpers для:
- проверки, можно ли очищать выбранный день
- очистки `workMs` по `dateKey`
- определения, нужно ли при старте таймера форсировать `work` override

Это даст тестируемость без раннего архитектурного рефакторинга.

### Блок B. Очистка дня как локальная операция renderer state

Так как данные дня сейчас живут в `localStorage` через renderer:
- очистка дня может быть реализована в renderer без нового сложного IPC
- после изменения должен вызываться обычный persist flow

Минимально приемлемая семантика:
- удалить запись `logs[key]`, если `workMs <= 0` после очистки
- либо привести запись к `{ workMs: 0 }`, если это проще для текущего кода

Предпочтительно:
- удалять пустую запись, чтобы не плодить лишние нулевые лог-записи

### Блок C. Автоперевод выходного в рабочий внутри `toggleRun()`

Рекомендуемое место:
- логика старта таймера

При старте:
1. вычислить сегодняшнюю дату
2. проверить `isDayOff(today)`
3. если день выходной, записать `dayOverrides[today] = "work"`
4. затем включить таймер

Важно:
- правило должно применяться только на переходе `stopped -> running`
- на `running -> stopped` статус дня не откатывается автоматически

### Блок D. Явный пересчёт streak после day actions

Любое действие из фазы:
- очистка дня
- смена day override
- старт таймера, меняющий выходной в рабочий

обязано завершаться единым обновлением:
- `renderCore()`
- либо эквивалентным набором вызовов, который гарантированно обновляет streak и календарь

## Ручные сценарии проверки

### Сценарий 1. Очистка прошлого дня

1. Иметь несколько дней с накопленным временем.
2. Выбрать прошлый день в календаре.
3. Выполнить очистку дня.
4. Подтвердить действие.

Ожидаемый результат:
- очищается только выбранный день
- остальные дни остаются без изменений
- streak и выбранная карточка обновляются сразу

### Сценарий 2. Попытка очистить сегодняшний день при активном таймере

1. Запустить таймер сегодня.
2. Оставить выбранным сегодняшний день.
3. Нажать `очистить день`.

Ожидаемый результат:
- очистка не выполняется
- пользователь получает понятное сообщение
- текущий таймер продолжает работать без повреждения данных

### Сценарий 3. Старт таймера в выходной

1. Выбрать конфигурацию, где сегодня считается выходным.
2. Запустить таймер.

Ожидаемый результат:
- сегодня автоматически становится рабочим днём
- в `dayOverrides` появляется `work` для сегодняшней даты
- streak и календарь обновляются сразу

### Сценарий 4. Историческая правка выходного

1. Иметь серию рабочих дней.
2. Задним числом отметить один из промежуточных дней как `off`.
3. Проверить streak.
4. Вернуть день обратно в рабочий.
5. Проверить streak ещё раз.

Ожидаемый результат:
- при `off` день исключается из streak-барьера
- при возврате в рабочий streak считается заново по обычным правилам

## Критерии приёмки

Фаза считается завершённой, если одновременно выполнены все условия:

- можно очистить конкретный день без полного сброса приложения
- очистка дня требует подтверждения
- очистка активного текущего дня не выполняется
- старт таймера в выходной автоматически переводит сегодняшний день в рабочий
- исторические правки выходных сразу пересчитывают streak
- добавлены тесты на исторические overrides и конфликт `workMs + off`
- `npm run lint` проходит
- `npm test` проходит
- если `tsconfig.json` отсутствует, отсутствие type-check явно фиксируется, а не игнорируется

## Риски и меры

### Риск 1. Очистка дня случайно начнёт менять статус дня

Мера:
- в коде развести операции "очистить время" и "изменить статус дня"
- не менять `dayOverrides` внутри clear-day flow

### Риск 2. Автоперевод выходного в рабочий будет срабатывать не на тот день

Мера:
- правило должно работать только для сегодняшней даты
- не использовать `selectedDate` как источник истины для этого сценария

### Риск 3. Исторические правки дадут непредсказуемый streak

Мера:
- зафиксировать явные правила приоритета `off` и `work`
- покрыть их unit-тестами до окончания фазы

### Риск 4. UI панели выбранного дня станет перегруженным

Мера:
- делать минимальное расширение текущей панели
- при нехватке места приоритизировать действия над декоративной симметрией

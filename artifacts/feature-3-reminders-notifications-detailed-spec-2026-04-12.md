## Название функции

Feature 3. Напоминания и уведомления

## Суть функции

Функция "Напоминания и уведомления" добавляет поверх обычного трекера времени лёгкий reminder-layer, который:

- не меняет саму модель учёта времени
- не превращает приложение в отдельный pomodoro-таймер
- а даёт мягкие сигналы во время работы через звук и системные уведомления

В рамках MVP это означает:
- таймер продолжает работать как обычный stopwatch
- поверх него можно включить режим напоминаний `25/5`
- во время активной сессии приложение в нужные моменты подаёт лёгкий звуковой сигнал и/или показывает Windows notification
- если desktop action buttons окажутся надёжными, они используются как дополнительный control layer; если нет, базовая функция всё равно считается рабочей без них

## Почему функция вводится

После Фаз `1-4` и `6` приложение уже умеет:
- стабильно учитывать рабочее время
- хранить интервалы
- показывать day details
- жить как desktop tray-приложение

Но пока у пользователя нет мягкого внешнего ритма работы:
- приложение фиксирует время, но не напоминает про рабочие циклы
- нет ненавязчивого сигнала "25 минут прошло"
- нет напоминания "5 минут отдыха закончились", если пользователь использует такой режим самодисциплины

Функция нужна, чтобы:
- добавить лёгкий ритм работы без переписывания трекера в полноценный pomodoro engine
- улучшить ежедневное использование приложения в фоне и в tray
- закрыть пользовательские заметки про `25/5` и Windows notifications без раздувания scope

## Предусловия функции

Feature 3 должна реализовываться только поверх уже принятой базы после интервалов и desktop polish.

Это означает:
- interval model уже существует и принята
- обычный start/stop timer flow уже стабилен
- hide-to-tray и tray lifecycle уже предсказуемы
- settings persistence уже умеет переживать новые поля
- main/preload/renderer уже готовы к аккуратному multi-surface поведению

Если timer flow, tray behavior или settings persistence ещё нестабильны, reminder-layer не считается готовым к реализации.

## Цель функции

После реализации функции пользователь должен уметь:

- включить или выключить reminder mode
- выбрать, нужны ли ему звуки, системные уведомления или оба канала сразу
- запустить обычный таймер и получить лёгкий сигнал через `25` минут
- получить второй сигнал через `5` минут после этого, не ломая сам учёт времени
- свернуть приложение в tray и всё равно получать напоминания
- при поддержке платформы использовать простые действия из системного уведомления, не открывая основное окно

## Что входит в функцию

- reminder mode поверх обычного таймера
- минимальный профиль напоминаний `25/5`
- лёгкие звуковые сигналы
- Windows notifications для reminder-событий
- настройки reminder-layer в приложении
- корректная работа напоминаний при hide-to-tray
- сброс reminder-cycle при остановке таймера
- явное поведение при sleep/resume и при долгом фоне
- опциональные action buttons в системном уведомлении, только если они реально надёжны на целевой Windows-связке

## Что не входит в функцию

- новый отдельный pomodoro-таймер с собственными состояниями `work/break/paused`
- изменение базовой доменной модели интервалов и учёта времени
- автоматическая остановка или пауза таймера по наступлению `25` или `5` минут
- полноценный scheduler "напомни мне начать работать через час"
- облачные push-уведомления
- кастомный выбор собственных мелодий пользователем в MVP
- тонкая настройка произвольных циклов `52/17`, `90/20` и т.п. в MVP
- отдельный центр уведомлений внутри приложения

## Ограничения функции

- приложение остаётся полностью оффлайн
- reminder-layer не должен менять semantics текущего таймера
- нельзя добавлять новые npm-зависимости без отдельного согласования
- нельзя блокировать UI ради расчёта следующего reminder-события
- нельзя дублировать business logic таймера отдельным "почти таким же" циклом
- нельзя делать reminder scheduling зависимым от видимости основного окна
- нельзя создавать новый persisted format только ради runtime reminder-состояния
- нельзя превращать feature в новый тяжёлый subsystem с десятком режимов и экранов

## Выбранные трактовки спорных решений

### 1. Это не pomodoro engine, а reminder-layer поверх stopwatch

Принятое правило:
- таймер по-прежнему считает обычное рабочее время
- напоминания не создают новую persisted state machine `work/break/pause`

Следствие:
- сигнал на `25` минуте не останавливает таймер
- сигнал на `30` минуте не запускает таймер заново, потому что он и не останавливался

### 2. MVP поддерживает один осознанный профиль: `25/5`

Принятое правило:
- в первой версии есть только:
  - `Off`
  - `25/5`

Пояснение:
- это прямо соответствует заметкам
- это минимально достаточный scope без UI для конструктора циклов

### 3. Напоминания считаются от старта текущей активной сессии

Принятое правило:
- baseline reminder-cycle начинается в момент старта текущего running interval
- остановка таймера завершает и рабочую сессию, и текущий reminder-cycle
- новый старт таймера создаёт новый цикл с нуля

Пояснение:
- это самый предсказуемый вариант
- не нужно переносить "остаток помодоро" между разными интервалами

### 4. Reminder-layer не работает в историческом режиме просмотра дня

Принятое правило:
- если пользователь выбрал прошлый день и видит серый historical timer, никаких напоминаний для этого режима нет
- reminders завязаны только на реальный running timer для `today`

### 5. Sleep/resume не должно порождать лавину просроченных уведомлений

Принятое правило:
- после resume приложение пересчитывает текущее elapsed time
- если во время сна были пропущены reminder milestones, допускается максимум одно догоняющее уведомление по текущей актуальной фазе
- приложение не должно выстреливать пачкой из всех пропущенных событий подряд

### 6. Системные notification actions являются условным расширением, а не ядром MVP

Принятое правило:
- базовая функция считается реализованной, если звуки и обычные уведомления работают стабильно
- кнопки действий в Windows notification добавляются только если Electron/Windows API позволяют сделать это надёжно

Пояснение:
- заметка `5` про запуск таймера из уведомления полезна, но она вторична по отношению к стабильной reminder-механике
- нельзя обещать desktop action flow, если платформа делает его хрупким

### 7. В MVP пользователь настраивает каналы напоминаний, но не аудиоредактор

Принятое правило:
- в настройках есть минимум:
  - режим напоминаний
  - `sound on/off`
  - `notifications on/off`

Не входит:
- выбор своей мелодии
- громкость по каналам
- отдельные звуки для work/break из внешних файлов пользователя

## Целевой пользовательский результат

После Feature 3 пользователь должен ощущать, что:

- приложение не только считает время, но и мягко ведёт рабочий ритм
- сигналы не мешают, а помогают
- reminder-layer работает даже если окно скрыто в tray
- напоминания не ломают интервалы, streak, totals, import/export и остальную логику

## Функциональные требования

## FR-1. Persisted reminder settings

В пользовательских настройках должны появиться новые поля reminder-layer.

Минимально ожидаемая модель:

```json
{
  "reminders": {
    "mode": "off",
    "soundEnabled": true,
    "notificationsEnabled": true
  }
}
```

Требования:
- старые сохранения без `reminders` продолжают работать
- invalid значения санитизируются к безопасным defaults
- reminder settings переживают перезапуск приложения

## FR-2. Reminder-cycle работает только при активном таймере

Если таймер:
- не запущен, никаких reminder-событий не должно происходить
- запущен, отсчёт milestones начинается от времени старта текущего running interval

Остановка таймера:
- завершает текущий reminder-cycle
- сбрасывает ожидание следующего сигнала

## FR-3. Профиль `25/5` даёт рабочий сигнал на 25-й минуте

При активном режиме `25/5`:
- на 25-й минуте с момента старта текущей сессии пользователь получает reminder-событие

Reminder-событие должно уметь включать:
- звук
- системное уведомление
- оба канала сразу

в зависимости от настроек пользователя.

## FR-4. Профиль `25/5` даёт второй сигнал через 5 минут после work cue

При активном режиме `25/5`:
- через 5 минут после сигнала `25` минут пользователь получает второй reminder-сигнал

Смысл второго сигнала:
- напомнить, что условная `5`-минутная пауза закончилась

При этом:
- таймер продолжает идти как обычный рабочий таймер
- приложение не переключается в отдельный break-state

## FR-5. Дальнейшие циклы повторяются предсказуемо

Если пользователь не остановил таймер, цикл продолжается дальше по тому же паттерну:
- `25` минут до work cue
- `5` минут до next cue
- затем снова `25`
- затем снова `5`

Проще говоря:
- reminder schedule повторяется поверх непрерывной running-session
- но не меняет её доменную семантику

## FR-6. Напоминания работают при hide-to-tray

Если окно скрыто, но приложение живо в tray и таймер активен:
- reminder-layer продолжает работать
- звук и/или системное уведомление всё равно срабатывают

Не допускается зависимость reminders от того, открыт ли основной renderer на экране.

## FR-7. Системные уведомления не должны ломать таймер

Notification dispatch:
- не должен менять running/stopped state сам по себе
- не должен создавать новые intervals
- не должен приводить к двойному start/stop из-за гонок IPC

Любое действие из уведомления должно проходить через явный IPC boundary и использовать уже существующий безопасный timer flow.

## FR-8. Notification actions допускаются только как thin-control layer

Если платформа позволяет стабильно поддержать кнопки действий, минимально допустимый набор:
- `Show`
- `Pause`

Условно допустимо:
- `Start`, но только если для него есть надёжный сценарий показа уведомления в idle-state

Не требуется:
- отдельный `Resume`
- отдельные действия для break-state, которого в MVP нет

## FR-9. Reminder-layer корректно ведёт себя при sleep/resume и долгом фоне

После wake/resume:
- reminder scheduler обязан пересчитать elapsed time
- не должен спамить всеми пропущенными сигналами
- должен перейти к текущему актуальному milestone корректно и предсказуемо

## FR-10. Reminder-layer не меняет export/import semantics

Feature 3 не должна требовать новой тяжёлой миграции persisted data.

Допускается:
- расширение `settings`

Не допускается:
- новый persisted event log ради reminder-runtime
- изменение interval entries только ради сигналов

## Ожидаемое техническое решение

## 1. Отдельный reminder domain layer

Reminder logic нельзя размазывать условиями по `app.js`.

Минимально ожидается отдельный модуль, например:
- `src/renderer/reminder-core.js`

Ответственность:
- sanitize reminder settings
- расчёт reminder milestones
- определение следующего события по elapsed time
- защита от повторного firing одного и того же milestone
- поведение на reset / stop / resume

## 2. Reminder runtime state остаётся runtime-only

Нужно развести:
- persisted user settings
- runtime scheduler state

Persisted:
- `mode`
- `soundEnabled`
- `notificationsEnabled`

Runtime-only:
- последний уже отданный milestone
- timestamp следующей локальной проверки
- состояние текущего активного reminder-cycle

Это нужно, чтобы:
- не раздувать persisted format
- не тащить эфемерное состояние в backup/import/export

## 3. System notification dispatch проходит через main/preload boundary

Если используются Windows notifications:
- renderer не должен напрямую становиться слоем платформенных уведомлений
- нужен явный IPC, например:
  - `reminders:notify`
  - `reminders:action`

Main-side слой отвечает за:
- показ системного уведомления
- обработку action callbacks, если они реально поддерживаются
- безопасный возврат команд в основной timer flow

## 4. Sound assets должны быть packaged-safe

Звуковые файлы, если они добавляются, должны:
- жить в runtime-доступном месте
- одинаково находиться в dev run и packaged build
- не зависеть от хрупких абсолютных путей

Подход:
- короткие встроенные reminder-assets
- явный helper разрешения asset paths

## 5. Scheduler должен быть cheap и не плодить гонки

Reminder scheduling не должен:
- создавать второй тяжёлый interval-loop поверх секундного timer UI без необходимости
- зависеть от частых перерисовок интерфейса

Минимально допустимый подход:
- использовать текущую timer-runtime информацию
- пересчитывать milestone cheap-функцией
- гарантировать, что один milestone не отрабатывает многократно

## Ожидаемые файлы изменений

Минимально ожидаемый набор:
- `src/renderer/app.js`
- `src/renderer/index.html`
- `src/renderer/styles.css`
- `src/renderer/tracker-core.js`
- новый reminder domain module, например `src/renderer/reminder-core.js`
- `src/preload.js`
- `src/main.js`
- возможно новый каталог runtime assets, например `src/assets/sounds/*`
- тесты для reminder domain logic
- обновление `ARCHITECTURE.md`

С высокой вероятностью:
- `test/tracker-core.test.js` или отдельный `test/reminder-core.test.js`

Не ожидаются без отдельной причины:
- `tracker-storage.js` за пределами минимального расширения settings boundary
- новая миграция persisted format только ради runtime scheduler state

## Этапы реализации внутри функции

### Этап A. Reminder settings и domain primitives

- ввести persisted reminder settings
- добавить sanitize/default logic
- вынести расчёт reminder milestones в отдельный модуль

### Этап B. Runtime scheduler

- привязать scheduler к running timer
- обеспечить reset на stop/start
- обработать sleep/resume и background scenarios

### Этап C. Sound и notification delivery

- подключить звук
- подключить системные уведомления
- провести dev/package-safe asset resolution

### Этап D. Settings UI и tray-safe UX

- добавить controls в настройки
- проверить работу при hide-to-tray
- убедиться, что feature не ломает обычный desktop flow

### Этап E. Optional notification actions

- проверить реальную поддержку action buttons
- если платформа даёт стабильный поток, добавить thin-control actions
- если нет, зафиксировать plain-notification fallback как конечное MVP-решение

## Автотесты

Минимально ожидаемое покрытие:

1. Settings sanitize
- отсутствие `reminders` в старом state даёт валидный default
- invalid `mode` откатывается в `off`

2. First milestone
- при active session reminder-core корректно определяет событие `25m`

3. Second milestone
- после `25m` корректно определяется событие `30m`

4. Repeating cycle
- далее цикл повторяется предсказуемо без дрейфа milestone ordering

5. Stop reset
- остановка таймера сбрасывает reminder-cycle

6. No duplicate firing
- один и тот же milestone не может сработать дважды при повторных тиках

7. Sleep/resume catch-up
- после резкого скачка elapsed time логика не генерирует бурст из всех пропущенных событий

8. Historical mode isolation
- historical selected day не порождает reminders сам по себе

## Ручные сценарии

Обязательный smoke перед принятием:

1. Включить `25/5`, оставить звук и уведомления включёнными, запустить таймер, убедиться что reminder-layer активировался без ошибок UI.
2. Проверить фактическую доставку первого сигнала на рабочем milestone.
3. Проверить доставку второго сигнала через `5` минут после него.
4. Свернуть приложение в tray и убедиться, что сигнал всё равно приходит.
5. Остановить таймер до очередного milestone и убедиться, что дальнейший сигнал не приходит.
6. Снова запустить таймер и убедиться, что цикл стартовал заново с нуля.
7. Отключить звук, оставить уведомления, убедиться что остаётся только notification-channel.
8. Отключить уведомления, оставить звук, убедиться что остаётся только sound-channel.
9. Усыпить/разбудить систему или воспроизвести длинную паузу и убедиться, что нет пачки из пропущенных уведомлений.
10. Если реализованы action buttons, проверить `Show` и `Pause` из системного уведомления.

Примечание:
- для ручной проверки допустимо использовать ускоренный dev-only test harness или временно сокращённые milestone-константы, но production-значения `25/5` не должны меняться в итоговой сборке

## Критерии приёмки

Функция считается принятой, если:

- reminder mode можно включить и выключить как обычную настройку
- `25/5` работает как overlay поверх обычного таймера, а не как новый timer engine
- пользователь получает сигналы в ожидаемые моменты
- reminders продолжают работать при hide-to-tray
- остановка таймера сбрасывает цикл предсказуемо
- sleep/resume не создаёт спам из уведомлений
- reminder-layer не меняет totals, intervals, streak и остальную доменную логику
- dev run и packaged build одинаково умеют доставлять звук и/или уведомления
- если action buttons нестабильны, функция остаётся принятой на plain-notification fallback без ложных обещаний

## Риски и меры

### Риск 1. Feature расползётся в отдельный pomodoro-продукт

Мера:
- заранее зафиксировать, что это reminder-layer, а не новый timer domain
- не вводить persisted work/break states в MVP

### Риск 2. Action buttons Windows notification окажутся нестабильными

Мера:
- сделать их условным расширением
- базовую ценность feature строить на sound + plain notifications

### Риск 3. Reminder scheduling начнёт дублировать timer runtime и разъедется с ним

Мера:
- вынести расчёт milestones в отдельный чистый модуль
- использовать уже существующий источник истины по timer state

### Риск 4. После сна системы пользователь получит пачку старых сигналов

Мера:
- делать catch-up не списком всех пропущенных milestones, а через вычисление актуального состояния цикла

### Риск 5. Аудио и notification assets будут работать только в dev

Мера:
- сразу проектировать packaged-safe asset resolution
- считать packaged smoke обязательной частью приёмки

## Итоговая трактовка функции

Feature 3 "Напоминания и уведомления" в рамках этого ТЗ трактуется как:

- лёгкий слой ритма работы поверх уже существующего трекера времени
- оффлайн-функция без облака и без отдельного центра уведомлений
- минималистичный reminder-mode `25/5`
- с акцентом на стабильность, а не на максимальное количество desktop-фич

Это не новый pomodoro-продукт и не новая модель времени.
Это аккуратное расширение текущего трекера, которое делает ежедневную работу чуть более управляемой и заметной.

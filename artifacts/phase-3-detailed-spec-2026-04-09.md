## Название фазы

Фаза 3. Подготовка архитектуры под интервалы

## Цель фазы

Создать техническую основу для функций, которые невозможно безопасно развивать на текущей модели дневных сумм:

- перейти от хранения только `workMs` по дням к модели, способной хранить рабочие интервалы

- развести внутри renderer ответственность между persistence, timer domain и UI wiring
- сохранить текущее пользовательское поведение, пока сами интервалы и ручная правка ещё не показаны в интерфейсе

## Выбранные трактовки спорных решений

### 1. Фаза 3 считается внутренним рефакторингом, а не новой пользовательской фичей

Принятое правило:
- после завершения Фазы 3 поведение приложения для текущих сценариев должно остаться прежним

Следствие:
- если по итогам фазы пользователь не видит новых экранов, это нормально
- ценность фазы в том, что Фаза 4 и Фаза 5 делаются уже не поверх хрупкой модели `logs[dateKey].workMs`

### 2. Новая модель должна иметь один источник истины по времени

Принятое правило:
- persisted данные не должны хранить независимую пару `workMs` и `intervals`, которые могут разойтись

Выбранный подход:
- источником истины становится список записей дня
- суммарное время дня вычисляется через helper/selectors, а не хранится как независимое редактируемое поле

Пояснение:
- это уменьшает риск рассинхрона между календарём, streak, аналитикой и будущей ручной правкой

### 3. Старые дневные суммы нельзя превращать в фальшивые интервалы с выдуманным временем суток

Принятое правило:
- при миграции старых данных нельзя выдумывать `09:00-15:00` или любое другое несуществующее время

Выбранный подход:
- старый дневной итог мигрирует в специальную запись вида `legacy-total`
- она хранит только длительность, но не притворяется реальным интервалом

Пояснение:
- это честно по отношению к данным пользователя
- это позволяет в будущем аналитике явно отличать реальные интервалы от исторических агрегатов

### 4. Реальные интервалы должны быть привязаны к локальным календарным дням

Принятое правило:
- интервал не должен пересекать полночь внутри одной записи

Следствие:
- если живая сессия пересекает границу дня, она должна раскладываться на две и более дневные записи
- это сохраняет уже существующий инвариант календаря и streak

### 5. Cold start остаётся нормализованным

Принятое правило:
- Фаза 3 не возвращает restore-running semantics

Следствие:
- даже если persisted/runtime слой становится сложнее, обычный холодный запуск всё равно поднимает приложение с остановленным таймером
- активная текущая сессия остаётся runtime-понятием, а не состоянием для восстановления после перезапуска

### 6. Rollback после миграции поддерживается только как best-effort

Принятое правило:
- старый формат нельзя обещать как полностью двусторонне совместимый после появления новых interval-данных

Выбранный подход:
- приложение начинает читать и писать новый формат `v8`
- старые ключи `v7...v1` продолжают читаться как источник миграции
- старый key не должен удаляться в момент чтения

Пояснение:
- это даёт безопасный старт миграции
- но если пользователь уже наработал новые interval-данные в `v8`, возврат на старую сборку `1.1.2` не сможет прочитать их полностью
- это нужно явно считать допустимым ограничением фазы

## Целевая модель данных

Новая persisted версия должна заменить модель `logs` на модель дневных записей.

Минимально ожидаемая структура:

```json
{
  "version": 8,
  "days": {
    "2026-04-09": {
      "entries": [
        {
          "type": "interval",
          "startMs": 1775680800000,
          "endMs": 1775684400000,
          "source": "timer"
        },
        {
          "type": "legacy-total",
          "durationMs": 5400000,
          "source": "migrated-v7"
        }
      ]
    }
  },
  "settings": {
    "dailyTargetHours": 6,
    "weekendDays": [0, 6],
    "dayOverrides": {},
    "language": "ru",
    "dateFormat": "localized",
    "weekStart": "monday",
    "autostart": false,
    "autoBackup": false
  },
  "timerState": {
    "isRunning": false
  }
}
```

Обязательные свойства модели:
- day key остаётся в формате `YYYY-MM-DD`
- `interval.startMs` и `interval.endMs` задаются в `Unix epoch milliseconds`
- для `interval` должно выполняться `endMs > startMs`
- `legacy-total.durationMs` должен быть `> 0`
- пустой день не хранится
- вычисление суммы дня идёт через helper, который суммирует все записи дня

## Инварианты новой модели

- ни одна запись `interval` не может пересекать полночь
- сумма дня считается только из записей дня, а не из параллельного поля-кэша
- `legacy-total` допускается только для мигрированных или импортированных агрегированных данных старого формата
- новые данные, записанные самим таймером после Фазы 3, должны создаваться как `interval`, а не как `legacy-total`
- streak и календарь должны брать рабочее время через единый selector поверх новой модели

## Функциональные требования

## FR-1. Новый persisted format

В проекте должна появиться новая версия persisted state:
- текущая `v7` заменяется на `v8`
- `STORAGE_KEY` и `PERSISTED_STATE_VERSION` должны быть согласованы

Обязательное свойство:
- новая структура должна быть пригодна для хранения реальных интервалов, а не только дневных сумм

## FR-2. Автоматическая миграция из `v7`

Если в хранилище пользователя есть данные старого формата:
- приложение обязано автоматически их прочитать
- преобразовать в runtime/persisted state нового формата
- сохранить суммарное время по всем дням без потерь

Правило миграции:
- каждый старый `logs[dateKey].workMs > 0` превращается в день с одной записью `legacy-total`
- `logs[dateKey].workMs <= 0` не должен создавать пустой день
- все настройки и day overrides должны переноситься без смысловых изменений

## FR-3. Импорт старых и новых backup-файлов

Import pipeline должен уметь:
- читать старые backup snapshot'ы со структурой `logs`
- читать новые snapshot'ы со структурой `days`
- нормализовать оба варианта в единый runtime формат

Export pipeline должен:
- экспортировать только новый формат после завершения Фазы 3

## FR-4. Разделение renderer-ответственности

После Фазы 3 в renderer должны быть разведены как минимум три уровня:

- persistence boundary
- timer domain
- UI wiring/rendering

Это означает:
- `app.js` больше не отвечает напрямую за чтение и запись `localStorage`
- `app.js` больше не содержит алгоритмическую логику разрезания времени по дням
- `app.js` работает через вызовы domain/persistence helper'ов

## FR-5. Timer path создаёт interval-записи

Новая живая сессия таймера должна записываться не как прямое добавление в `workMs`, а как interval-based mutation.

Требование:
- при `flushTick` и связанных сценариях доменный слой должен уметь раскладывать живое время на дневные записи
- если сессия пересекает полночь, должны создаваться отдельные interval-записи по дням

## FR-6. Поведение Фазы 1 и Фазы 2 сохраняется

На новой модели не должны сломаться:
- cold start без автозапуска таймера
- корректная синхронизация autostart
- clear data
- clear selected day
- `day off` и `work` override
- streak по историческим выходным

## FR-7. Старая логика streak должна работать поверх новой модели

`calculateCurrentStreak` и связанная UI-логика должны использовать общий selector суммарного времени дня, а не старую зависимость от `logs[dateKey].workMs`.

Требование:
- migrated `legacy-total` должен участвовать в дневной сумме так же, как прежний `workMs`
- `dayOverrides.off` по-прежнему исключает день из streak независимо от наличия времени

## FR-8. Явная стратегия rollback и coexistence

Фаза должна явно описать и реализовать безопасное coexistence-поведение:
- новый билд сначала читает `v8`, затем старые ключи как legacy source
- старые ключи не удаляются автоматически только от самого факта чтения
- clear data удаляет все версии ключей

Ограничение rollback:
- возврат на старую сборку после появления новых interval-данных официально считается only best-effort

## FR-9. Тестируемость новой модели

Новая архитектура должна вводиться так, чтобы критические вещи покрывались unit-тестами:
- миграция `v7 -> v8`
- суммирование дня из `interval` и `legacy-total`
- разрезание interval по полуночи
- import старого backup'а
- сохранение поведения streak на migrated-данных

## Предлагаемое техническое решение

### Блок A. Новый domain-слой в `tracker-core.js`

`tracker-core.js` должен стать местом для pure logic:
- нормализация persisted state `v8`
- миграция старых форматов в `v8`
- day/entry selectors
- суммирование времени дня
- расчёт streak поверх новых selectors
- helper'ы для добавления interval и legacy-total entry

Требование:
- в этом модуле не должно быть `localStorage`, DOM и Electron API

### Блок B. Выделенный persistence boundary

Нужен отдельный renderer-модуль для:
- списка storage keys
- чтения последней поддерживаемой версии данных
- сохранения snapshot в `localStorage`
- удаления всех legacy keys
- адаптации import/export snapshot'ов

Ожидаемое имя:
- `src/renderer/tracker-storage.js` или эквивалент с одной ответственностью

### Блок C. Выделенный timer domain

Нужен отдельный модуль, который отвечает за:
- runtime timer state
- start/stop/flush
- pause/resume
- разрезание живого времени по границам дней
- запись interval-мутаторов в day model

Ожидаемое имя:
- `src/renderer/tracker-timer.js` или эквивалент

### Блок D. `app.js` как wiring-layer

После завершения фазы `app.js` должен остаться местом для:
- DOM refs
- i18n и текстов интерфейса
- render-функций
- event handlers
- вызовов domain/persistence API

Что должно уйти из `app.js`:
- прямое знание формата persisted JSON
- прямой `localStorage` read/write
- расчёт разбиения времени по дням
- детали миграции между версиями данных

### Блок E. Backup contract

Так как backup уже передаётся как snapshot:
- `main` и `preload` должны остаться transport-only
- бизнес-логика преобразования старого и нового snapshot остаётся в renderer domain/persistence слоях

Это минимизирует риск регрессий по Electron boundary.

- `app.js` больше не содержит прямой `localStorage` persistence и interval-splitting logic
- import/export работают и для старых, и для новых snapshot'ов
- Фазы 1-2 не ломаются на новой архитектуре

## Риски и меры

Риск:
- старая дневная сумма не содержит реальных часов начала/конца

Мера:
- мигрировать такие данные только в `legacy-total`, без выдумывания фальшивых interval timestamps

Риск:
- рефакторинг сломает рабочие сценарии UI без заметной новой пользовательской ценности

Мера:
- выполнять фазу по этапам с поведением, совместимым после каждого шага
- держать unit-тесты на domain и migration до переподключения UI

Риск:
- после миграции старый билд `1.1.2` не сможет прочитать новые interval-данные

Мера:
- считать rollback only best-effort
- не удалять старый key при одном только чтении
- отдельно зафиксировать это ограничение в release notes к `1.2.0`

Риск:
- `app.js` останется большим, но просто с другими зависимостями

Мера:
- при приёмке отдельно проверять не только прохождение тестов, но и фактический срез ответственности файла

## Итог фазы

Результатом Фазы 3 должен стать не новый экран, а новая база:

- данные умеют хранить интервалы
- старые данные мигрируются без потери сумм
- timer path работает поверх interval model
- UI больше не привязан напрямую к persisted JSON старого формата

Только после этого безопасно переходить к Фазе 4:
- показ интервалов
- ручная правка времени
- подготовка агрегатов под аналитику

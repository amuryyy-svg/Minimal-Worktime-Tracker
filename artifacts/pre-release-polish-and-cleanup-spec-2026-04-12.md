## Цель

Привести проект к состоянию, в котором его можно безопасно собирать и выпускать как `1.2.0` без случайных хвостов, локального мусора, недоделанной упаковки и непроверенных критичных сценариев.

## Что входит в это ТЗ

- чистка рабочего дерева и git-гигиена
- комплектность release change set
- финальная доводка packaging metadata
- bump версии на `1.2.0`
- обязательная автоматическая и ручная верификация
- минимальная документационная синхронизация перед релизом

## Что не входит в это ТЗ

- новая продуктовая разработка
- новая аналитика, артефакты или reminder-layer
- переписывание архитектуры
- редизайн интерфейса
- новые npm-зависимости

## Наблюдаемые хвосты на текущий момент

По состоянию рабочего дерева сейчас уже видны конкретные релизные хвосты:

- есть untracked служебная папка `.codex/`
- есть untracked продуктовые assets в `src/assets/`
- есть untracked документы в `artifacts/`, которые нужно либо осознанно включить в репозиторий, либо осознанно не публиковать
- `package.json` всё ещё содержит версию `1.1.2`
- в `package.json` отсутствует поле `author`
- `Logo.png` изменён и должен быть отдельно проверен на намеренность этого diff
- `.gitignore` не игнорирует `.codex/`, то есть служебный каталог будет постоянно всплывать как локальный мусор

## Выбранная трактовка задачи

Это не "ещё одна фаза".
Это короткий pre-release hardening pass.

Его задача:
- не добавить новых возможностей
- а сделать так, чтобы в релиз ушли только нужные файлы, правильные метаданные, правильная версия и проверенная сборка

## Блок 1. Чистота рабочего дерева и git-гигиена

Нужно привести дерево к intentional state.

Обязательные задачи:
- решить судьбу `.codex/`
- если каталог не должен публиковаться, добавить его в `.gitignore`
- убедиться, что в релизный change set не попадает локальный служебный мусор
- проверить untracked файлы в `artifacts/` и `src/assets/`
- по каждому такому файлу принять явное решение: `входит в релизный репозиторий` или `не входит`
- добиться состояния, в котором перед релизом в `git status` нет случайных untracked файлов

Особое правило:
- нельзя оставлять ситуацию "код зависит от файла, который так и висит untracked"

## Блок 2. Комплектность релизного change set

Нужно проверить, что все реально используемые релизом файлы включены в финальный набор.

Обязательные задачи:
- включить tray assets из `src/assets/tray/*`, если именно они являются боевым runtime pipeline
- проверить, что `package.json -> build.files` действительно покрывает runtime assets
- проверить, что документы в `artifacts/` попадают в git только если вы хотите хранить ТЗ в репозитории
- отдельно проверить `Logo.png`

По `Logo.png` нужно принять одно из двух решений:
- если это осознанное изменение продуктового asset, оно входит в релиз и проходит визуальную проверку
- если это случайный локальный diff, его нельзя тащить в релиз "просто потому что файл изменён"

## Блок 3. Packaging metadata и release identity

Нужно закрыть минимальные метаданные выпуска.

Обязательные задачи:
- обновить версию в `package.json` с `1.1.2` до `1.2.0`
- добавить `author` в `package.json`
- проверить `productName`, `appId`, имена installer/portable artifacts на соответствие релизу
- убедиться, что сборка маркируется как `1.2.0`, а не как старая версия

Не требуется:
- менять package name
- менять build target
- проводить редизайн branding

## Блок 4. Документационная синхронизация

Нужно привести документацию к честному состоянию относительно текущего кода.

Обязательные задачи:
- убедиться, что `ARCHITECTURE.md` соответствует реально внесённым архитектурным изменениям
- решить, какие документы в `artifacts/` публикуются вместе с репозиторием
- не оставлять частично добавленные ТЗ: либо они входят в историю репозитория, либо исключаются осознанно

Минимально допустимый результат:
- архитектурный документ не врёт
- набор artifact-документов в репозитории выглядит намеренным, а не случайным

## Блок 5. Автоматическая верификация

Перед релизом обязательно пройти:

1. `npm run lint`
2. `npm test`
3. `npm run dist`

Дополнительно:
- если type-check по-прежнему не настроен, это должно быть явно зафиксировано, а не забыто

Нельзя выпускать релиз, если:
- автоматические проверки красные
- сборка собирается не под той версией
- после сборки остаются непонятые packaging regression issues

## Блок 6. Обязательный ручной smoke перед релизом

Нужен короткий, но реальный smoke именно на собранном Windows-билде.

Минимальный набор сценариев:

1. Приложение запускается как собранный билд.
2. Tray icon отображается корректно.
3. `minimize` уводит приложение в tray.
4. `close` не убивает приложение, а уводит в tray, если это текущая принятая логика.
5. Tray menu работает: `Show/Hide/Start/Pause/Exit`.
6. Тёмная тема переключается и переживает перезапуск.
7. Ручная правка дня с interval-данными работает.
8. Imported/legacy day корректно отображается и редактируется без поломки totals.
9. Нельзя некорректно редактировать текущий день при активной сессии, если это правило уже есть в коде.
10. `clear data` отрабатывает безопасно.
11. Import/export не ломаются после всех последних изменений.

Если релиз претендует включать ранее закрытые баги Фаз 1-2, дополнительно руками пройти:
- автозагрузка
- cold start после предыдущей активной сессии
- очистка отдельного дня

## Блок 7. Финальная release-чистка

Перед публикацией нужно сделать последний проход по change set.

Обязательные задачи:
- убрать случайные локальные файлы и служебные хвосты
- убедиться, что нет debug-артефактов и временных файлов
- убедиться, что `.gitignore` покрывает локальные каталоги, которые не должны регулярно всплывать
- собрать короткий release summary: что вошло в `1.2.0`

Минимально ожидаемое содержимое release summary:
- ключевые изменения пользователя
- важные desktop/UI улучшения
- ограничения и что сознательно отложено

## Порядок выполнения

1. Сначала очистить рабочее дерево и определить состав релизного change set.
2. Затем добить packaging metadata и bump версии.
3. После этого пройти автоматическую верификацию.
4. Затем пройти ручной smoke на собранном билде.
5. Только после этого фиксировать итоговый релизный набор файлов.

## Критерии приёмки

Эта pre-release чистка считается завершённой, если:

- `git status` не содержит случайных untracked хвостов
- `.codex/` больше не мешает релизному дереву
- все реально используемые assets и документы либо осознанно включены, либо осознанно исключены
- версия в коде обновлена до `1.2.0`
- `author` и минимальные packaging metadata заполнены
- `npm run lint` проходит
- `npm test` проходит
- `npm run dist` проходит
- собранный Windows-билд проходит короткий ручной smoke
- release summary можно отдать в публикацию без дополнительных раскопок

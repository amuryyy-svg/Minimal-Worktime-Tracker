## Цель

Стабилизировать текущее приложение, закрыть подтверждённые баги поведения, затем безопасно расширить модель данных так, чтобы можно было добавить интервалы, аналитику и ручную корректировку времени без хрупких точечных костылей.

## Что нельзя сломать

- базовый сценарий: старт/стоп таймера, отображение текущего дня, календарь, streak, tray
- импорт и экспорт бэкапа
- автозапуск приложения как системная функция Windows
- совместимость с уже сохранёнными данными пользователя

## Общие правила выполнения

- Каждая фаза должна завершаться рабочей сборкой и проходить текущую верификацию.
- Нельзя начинать фазу с изменением persisted data format, пока не описана миграция и откат.
- Для задач `11-14` нельзя продолжать раздувать `src/renderer/app.js`; до этих задач нужно выделить отдельный слой состояния и таймера.
- Любые изменения по tray, autostart и restore state должны проверяться ручным сценарием в Electron, потому что существующие unit-тесты это не покрывают.

## Порядок фаз

### Фаза 1. Стабилизация критичных багов поведения

Цель:
- убрать баги, которые уже подтверждены по заметкам и затрагивают доверие к приложению

Задачи:
- исправить рассинхрон автозагрузки между `localStorage` и состоянием Windows
- исправить автозапуск таймера после холодного старта приложения
- воспроизвести и исправить ошибку общего сброса данных
- сделать сценарий сброса данных атомарным с понятным результатом для UI
- добавить минимальные проверки на startup state и clear data

Пункты из заметок:
- `1`
- `6`
- `15`

Рекомендуемый результат реализации:
- единый источник истины для autostart или явная двусторонняя синхронизация main ↔ renderer
- холодный запуск всегда поднимает приложение с остановленным таймером, если отдельно не введён режим восстановления
- clear data либо завершается полностью, либо явно возвращает ошибку в UI

Критерии приёмки:
- после включения автозагрузки и перезапуска приложения настройка остаётся согласованной в UI и в системе
- после закрытия приложения с активным таймером новый ручной запуск не стартует таймер сам
- общий сброс данных не падает и после него интерфейс показывает чистое состояние

### Фаза 2. Корректность работы по дням и streak

Цель:
- привести в порядок поведение календаря и логики выходных до начала крупной переработки модели данных

Задачи:
- добавить очистку выбранного дня с явным подтверждением
- определить и реализовать поведение запуска таймера в выходной день
- усилить логику streak для исторических правок выходных
- добавить тесты на `dayOverrides`, историческое помечание выходного и комбинации `workMs + off`
- проверить, что очистка дня и смена статуса дня не ломают streak и календарь

Пункты из заметок:
- `7`
- `16`
- `17`

Рекомендуемый результат реализации:
- у выбранного дня появляется отдельное действие очистки времени
- если пользователь стартует таймер в выходной, день явно переводится в рабочий по согласованному правилу
- историческое выставление выходного корректно пересчитывает streak

Критерии приёмки:
- можно очистить конкретный день без полного сброса приложения
- после ручной правки прошлых выходных streak пересчитывается предсказуемо
- запуск таймера в выходной не оставляет день в противоречивом состоянии

### Фаза 3. Подготовка архитектуры под интервалы

Цель:
- создать техническую основу для задач, которые не укладываются в текущую модель дневных сумм

Задачи:
- отделить persistence / timer domain / UI wiring внутри renderer
- сократить связанность `src/renderer/app.js`
- спроектировать новую структуру persisted state
- подготовить версионирование формата и миграцию данных
- сохранить обратную совместимость импорта старых бэкапов

Пункты из заметок:
- техническая база для `11`, `12`, `13`, `14`

Рекомендуемый результат реализации:
- новая версия persisted state хранит не только `workMs`, но и интервалы работы
- миграция из текущей версии не теряет накопленное время
- UI не зависит от деталей хранения данных напрямую

Критерии приёмки:
- существующие сохранения пользователя открываются без ручного вмешательства
- старые бэкапы импортируются
- после реорганизации кодовая база остаётся читаемой, а timer logic вынесен из монолитного UI-файла

### Фаза 4. Интервалы, ручная корректировка и аналитическая база

Цель:
- реализовать пользовательскую ценность, ради которой нужен redesign модели данных

Задачи:
- добавить фиксацию интервалов работы
- добавить возможность ручной правки рабочего времени
- определить модель редактирования: редактируем суммарное время, интервалы или оба представления
- показать интервалы в интерфейсе выбранного дня
- подготовить агрегаты для аналитики

Пункты из заметок:
- `11`
- `13`

Рекомендуемый результат реализации:
- пользователь видит не только сумму за день, но и список рабочих промежутков
- ручная правка не ломает внутреннюю структуру данных
- изменение времени отражается в календаре, streak и бэкапах

Критерии приёмки:
- можно добавить или исправить время дня без запуска таймера
- интервалы сохраняются после перезапуска
- экспорт / импорт корректно переносят новые данные

Пункты из заметок:
- `10`
- `12`

Рекомендуемый результат реализации:

- настройки либо остаются в overlay с улучшенным UX, либо выносятся в отдельное окно только если это действительно упрощает продукт

Критерии приёмки:
- аналитика открывается отдельно и не блокирует основное окно
- данные в аналитике совпадают с календарём и дневными карточками
- решение по окну настроек принято осознанно, а не как декоративный перенос

## Что не включать в первые этапы

- переписывание приложения с Electron на другой стек

- аналитику до появления интервалов
- отдельные окна без реальной необходимости в новых потоках данных
